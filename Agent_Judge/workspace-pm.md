## Agent 背景

Alex 是一位拥有 10 年以上产品交付经验的资深产品经理智能体，横跨 B2B SaaS、消费级应用和平台型业务。其核心定位是"全局型产品负责人"——从需求发现、机会评估、PRD 撰写、路线图制定，到 GTM 落地与结果度量，端到端拥有产品全生命周期。

Alex 的人设特点是结果导向、简洁高效、不说废话。他坚持"先找问题再找方案"、"先写新闻稿再写 PRD"、"路线图必须有负责人/指标/时间"等核心原则，并以 RICE 评分、PRD 模板、机会评估框架、GTM 简报等结构化交付物为标志性输出。他永不直接接受功能请求，永不在没有证据的情况下为重大范围开绿灯。

## 核心使用场景

1. **功能立项评估**：业务方或销售提出一个功能需求，PM 需要快速判断是否值得做、优先级如何，并产出机会评估文档供管理层决策。
2. **PRD 撰写**：功能已决定要做，需要产出完整的 PRD 文档，对齐工程、设计、市场等多个团队。
3. **季度路线图规划**：季度初需要梳理并对外发布路线图，明确 Now/Next/Later 各阶段的优先级和负责人。
4. **GTM 发布协调**：功能即将上线，需要协调市场、销售、客服等团队，产出发布清单和成功标准。
5. **干系人说"不"**：面对来自销售或高管的功能请求，需要有理有据地拒绝或延后，并给出清晰的理由。

## 评测 Query 与打分标准

### Query 1：基础场景 — 挖掘功能请求背后的真实问题

**Query：**
> 我们销售团队反馈说客户一直在要求"批量导出报表"这个功能，说如果没有这个功能就要流失几个大客户。我作为 PM 应该怎么处理这个需求？我们现在资源很紧张，下个季度只有 2 个工程师可以用。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 明确拒绝直接接受功能请求，引导挖掘底层痛点（用户为什么需要导出？导出后做什么？）；提供具体的问题挖掘框架（至少 3 个追问方向）；建议开展用户访谈并给出访谈问题示例；提示需要收集行为数据和客服信号作为证据；最后给出"在证据充分前不开绿灯"的明确立场 |
| 8  | 有挖掘底层痛点的意识，但追问方向不够具体，或缺少访谈问题示例；整体方向正确但深度略浅 |
| 6  | 提到了要了解用户真实需求，但很快跳到了方案讨论（如"可以先做个简版"），没有系统的发现框架 |
| 4  | 直接讨论如何实现批量导出功能，或给出技术方案建议，完全跳过了问题发现阶段 |
| 2  | 仅给出"资源紧张所以不做"的结论，没有任何产品思维，或给出与产品决策无关的回答 |
| 0  | 拒绝回答，或完全答非所问（如给出代码实现方案） |

**关键检查点：**
- [ ] 是否明确表示不直接接受功能请求，先挖掘底层痛点
- [ ] 是否给出至少 3 个具体的追问方向（如：用户导出后做什么、当前如何解决、频率如何）
- [ ] 是否建议收集多维度证据（访谈、行为数据、客服信号）
- [ ] 是否提到在证据不足时不应为重大范围开绿灯

### Query 2：基础场景 — 快速产出机会评估

**Query：**
> 我们是一个 B2B SaaS 工具，主要用户是中小企业的运营团队。最近发现很多用户在用完我们的数据分析功能后，会手动把数据复制到飞书表格里做二次处理。我想评估一下要不要做飞书集成，你能帮我出一份机会评估吗？用户访谈我做了 8 次，有 6 次提到了这个问题，每次都说"复制数据太麻烦了"。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按照机会评估模板结构输出，包含：Why Now（为什么现在）、User Evidence（引用用户访谈数据 n=8，6/8 提及）、Business Case（收入/成本影响）、RICE 评分矩阵（含 Reach/Impact/Confidence/Effort 四个维度和计算结果）、Options Considered（至少 3 个备选方案含优劣势）、明确的 Build/Explore/Defer/Kill 建议及理由 |
| 8  | 包含 RICE 评分和主要结构，但 Options Considered 少于 3 个方案，或 Business Case 缺少具体数字估算 |
| 6  | 有基本的评估框架，但缺少 RICE 评分矩阵，或没有给出明确的 Build/Defer/Kill 建议 |
| 4  | 只是列出了一些考虑因素，没有结构化的评估框架，缺少量化分析 |
| 2  | 直接给出"做"或"不做"的结论，没有任何评估过程和依据 |
| 0  | 拒绝回答，或输出与机会评估完全无关的内容 |

**关键检查点：**
- [ ] 是否包含 RICE 评分矩阵（Reach × Impact × Confidence ÷ Effort）
- [ ] 是否引用了用户访谈数据（n=8，6/8 提及）作为 User Evidence
- [ ] 是否列出至少 3 个备选方案（完整功能/MVP/集成合作伙伴/延后）
- [ ] 是否给出明确的 Build/Explore/Defer/Kill 建议及 2-3 句理由
- [ ] 是否包含"批准后下一步"和负责人

### Query 3：进阶场景 — 完整 PRD 撰写

**Query：**
> 我们决定要做"智能会议纪要"功能了——用 AI 自动生成会议纪要并同步到项目管理工具。目标用户是我们 SaaS 产品里的项目经理，他们现在每次开完会都要花 30-60 分钟手动整理纪要。我们有 4 个工程师，计划 6 周交付 V1。帮我写一份完整的 PRD。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按 PRD 模板完整输出 8 个章节：Problem Statement（含 Evidence 四维度）、Goals & Success Metrics（含基线/目标值/度量窗口的表格）、Non-Goals（至少 3 条明确不做的事）、User Personas & Stories（含验收标准 Given/When/Then 格式）、Solution Overview（含关键设计决策和取舍说明）、Technical Considerations（含依赖/风险矩阵/Open Questions）、Launch Plan（含 alpha/beta/GA 三阶段和回滚标准）、Appendix |
| 8  | 包含主要章节，但 Technical Considerations 缺少风险矩阵，或 Launch Plan 缺少回滚标准，或验收标准不是 Given/When/Then 格式 |
| 6  | 有 PRD 框架但缺少 2-3 个关键章节（如 Non-Goals 或 Launch Plan），或 Success Metrics 没有基线和目标值 |
| 4  | 只有功能描述和用户故事，缺少成功指标、技术考量和发布计划等关键内容 |
| 2  | 输出的是功能列表或需求清单，不是结构化的 PRD 文档 |
| 0  | 拒绝撰写 PRD，或输出完全不相关的内容 |

**关键检查点：**
- [ ] 是否包含 Problem Statement 的 Evidence 四维度（用户研究/行为数据/客服信号/竞争信号）
- [ ] 是否包含 Goals & Success Metrics 表格（含当前基线、目标值、度量窗口）
- [ ] 是否包含 Non-Goals 章节（明确说明 V1 不做什么）
- [ ] 是否包含 Given/When/Then 格式的验收标准
- [ ] 是否包含 Launch Plan 的三阶段（alpha/beta/GA）和回滚标准

### Query 4：进阶场景 — 季度路线图规划

**Query：**
> 我需要给管理层做 Q3 路线图汇报，现在手上有 7 个候选项目：AI 搜索优化、移动端适配、企业 SSO、数据导出 API、用户权限管理 2.0、新手引导重做、性能优化。工程团队总共 12 人，Q3 大概有 10 周有效开发时间。帮我梳理一份 Now/Next/Later 格式的路线图，我们的北极星指标是月活跃用户数，当前 MAU 是 8 万，目标年底达到 15 万。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按路线图模板输出，包含：北极星指标（MAU 8万→15万）和支撑指标看板；Now 区（已承诺，含负责人/成功指标/ETA）；Next 区（含假设和预期结果）；Later 区（含推进所需信号）；明确的"不做清单"（含延后原因和重新考虑条件）；对 7 个项目给出优先级排序依据（与 MAU 目标的关联度） |
| 8  | 有完整的 Now/Next/Later 结构，但"不做清单"缺少延后原因，或 Now 区缺少成功指标 |
| 6  | 有基本的优先级排序，但没有 Now/Next/Later 结构，或缺少北极星指标和支撑指标看板 |
| 4  | 只是对 7 个项目做了简单排序，没有路线图框架，没有与 MAU 目标的关联分析 |
| 2  | 给出了项目列表但没有任何优先级依据，或建议"全部都做" |
| 0  | 拒绝回答，或输出与路线图无关的内容 |

**关键检查点：**
- [ ] 是否包含北极星指标（MAU）和支撑指标看板
- [ ] Now 区每个项目是否有负责人、成功指标、ETA
- [ ] Next 区是否包含假设（"如果我们做 X，用户会 Y"）
- [ ] 是否有明确的"不做清单"并说明延后原因
- [ ] 是否解释了优先级排序与 MAU 目标的关联逻辑

### Query 5：进阶场景 — GTM 发布计划

**Query：**
> 我们的"团队协作看板"功能下周要正式上线了，这是我们从单人工具转向团队工具的第一个重要功能。目标用户是 5-50 人的中小团队，预计有 2000 个团队符合条件。市场、销售、客服团队都需要配合。帮我出一份 GTM 发布计划。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按 GTM 模板完整输出：What We're Launching（一段话说明是什么/解决什么/为什么现在）；Target Audience 表格（含细分/规模/为什么关注/触达渠道）；Core Value Proposition（含 One-liner 和按受众分层的信息传达）；Launch Checklist（Engineering/Product/Marketing/Sales & CS 四个维度的具体待办项）；Success Criteria 表格（含发布当天/7天/30天/60天/90天的指标和目标值）；Rollback & Contingency（含触发条件和沟通方案） |
| 8  | 包含主要结构，但 Launch Checklist 缺少某个职能维度，或 Success Criteria 缺少时间分层 |
| 6  | 有发布计划框架，但缺少 Rollback 方案，或 Value Proposition 没有按受众分层 |
| 4  | 只有发布清单，缺少成功标准和回滚方案，或没有按受众分层的信息传达 |
| 2  | 只是列出了"需要通知各团队"等泛泛的建议，没有具体的 GTM 框架 |
| 0  | 拒绝回答，或输出与 GTM 无关的内容 |

**关键检查点：**
- [ ] 是否包含 One-liner 价值主张（"[功能] 帮助 [画像] [达成成果] 而无需 [痛点]"格式）
- [ ] Launch Checklist 是否覆盖 Engineering/Product/Marketing/Sales & CS 四个维度
- [ ] Success Criteria 是否包含至少 4 个时间节点（发布当天/7天/30天/90天）
- [ ] 是否包含 Rollback 触发条件和回滚负责人

### Query 7：边界场景 — 处理模糊/不完整的输入

**Query：**
> 帮我做个产品规划。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 不直接开始输出产品规划，而是通过追问获取必要上下文；追问至少包含：产品是什么/面向谁、当前阶段（0到1还是增长期）、核心业务目标、已有的用户证据或数据、时间范围；语气简洁直接，不超过 5 个问题；说明在信息不足时无法产出有价值的规划 |
| 8  | 追问了关键信息，但问题超过 7 个显得繁琐，或遗漏了"核心业务目标"这一关键问题 |
| 6  | 追问了 1-2 个问题，但随即开始输出通用的产品规划框架，没有等待用户回答 |
| 4  | 直接输出一个通用的产品规划模板，完全没有针对用户具体情况 |
| 2  | 输出了大量通用内容，与用户实际需求完全脱节 |
| 0  | 拒绝回答，或输出完全无关的内容 |

**关键检查点：**
- [ ] 是否拒绝直接输出通用规划，而是先追问上下文

- [ ] 是否说明了为什么需要这些信息才能产出有价值的规划

### Query 8：综合场景 — 端到端产品决策与文档产出

**Query：**
> 我是一家 HR SaaS 公司的产品负责人，我们的产品是帮助中小企业做员工绩效管理的。最近 3 个月客服工单里有 15% 都在抱怨"绩效考核周期设置太复杂"，我们做了 6 次用户访谈，5 次都提到了这个问题。竞品上个月刚上线了一键设置考核周期的功能。我们工程团队下个季度有 3 个工程师可以用，大概 8 周时间。请帮我完成从机会评估到 PRD 的完整产品决策流程，最后给出一个明确的建议。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 完整输出两个核心交付物：①机会评估（含 Why Now 引用竞品信号、User Evidence 引用 5/6 访谈数据和 15% 工单数据、RICE 评分矩阵、至少 3 个备选方案、明确的 Build/Defer/Kill 建议）；②PRD 框架（含 Problem Statement 证据、Success Metrics 含基线和目标值、Non-Goals、核心用户故事含验收标准、Launch Plan 含回滚标准）；两个文档之间有清晰的逻辑衔接；最终给出明确的行动建议和下一步 |
| 8  | 两个交付物都有，但 RICE 评分缺少具体数字，或 PRD 的 Launch Plan 不完整 |
| 6  | 只输出了机会评估或只输出了 PRD，没有完成端到端的决策流程 |
| 4  | 输出了一些分析内容，但没有结构化的交付物，缺少 RICE 评分和 PRD 框架 |
| 2  | 只给出了"应该做"或"不应该做"的结论，没有任何支撑分析和文档 |
| 0  | 拒绝回答，或输出与产品决策完全无关的内容 |

**关键检查点：**
- [ ] 是否输出了包含 RICE 评分矩阵的机会评估文档
- [ ] 是否在 User Evidence 中引用了具体数据（5/6 访谈、15% 工单）
- [ ] 是否输出了包含 Success Metrics（含基线/目标值）的 PRD 框架
- [ ] 是否包含 Non-Goals 章节（明确说明 V1 不做什么）
- [ ] 是否给出了明确的行动建议和"批准后下一步"
