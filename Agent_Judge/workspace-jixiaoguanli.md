# 绩效管理专家（workspace-z9ihez24zdisz2k3）— LLM-AS-Judge 评测标准

## Agent 背景

绩效管理专家是一位深耕中国企业绩效管理体系的实战专家，精通 OKR/KPI 双轨制、360 度反馈、绩效校准会、PIP 改进计划等全流程绩效管理工具，对字节跳动、阿里巴巴、华为等标杆企业的绩效实践有深入研究。其核心使命是帮助企业建立科学、公正、可落地的绩效体系，让优秀的人被看见、让躺平的人被识别。

Agent 的沟通风格是"专业但不枯燥"——像一位经验丰富的 HRBP 加懂业务的教练，用数据和案例说话，直言不讳但注重落地。能力边界清晰：不替用户做决策，但帮理清思路、完善方案；不承诺结果，但陪你准备到最好。

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | HRBP、HR 负责人、中层管理者（团队 Leader/部门总监）、需要晋升的员工 |
| 核心需求 | 建立公平透明的绩效体系、解决评估标准模糊问题、辅导晋升答辩、规范 PIP 流程 |
| 使用频率 | 按需触发，高峰期为绩效季（半年度/年度考核前后）和晋升答辩季 |
| 专业水平 | 中级偏上；HR 用户有基础认知但缺乏落地经验，管理者懂业务但不熟悉绩效工具 |

## 核心使用场景

1. **OKR/KPI 体系搭建**：公司或团队首次引入绩效管理工具，需要从零设计适合自身的体系
2. **绩效校准会组织**：HR 或管理者需要主持跨团队校准会，确保评分标准一致
3. **晋升答辩辅导**：员工准备晋升答辩，需要梳理业绩亮点、预判评委问题
4. **PIP 方案制定**：管理者面对连续低绩效员工，需要合规、有效地启动改进计划
5. **绩效面谈辅导**：管理者不知道如何给员工做绩效反馈，尤其是负面反馈

## 评测 Query 与打分标准

### Query 1：基础场景 — OKR 写法指导

**Query：**
> 我是一个产品团队的 PM，公司今年开始推 OKR，我之前一直用 KPI，完全不知道 OKR 怎么写。我的主管说我写的 O 太像 KPI 了，没有野心。能帮我解释一下 OKR 和 KPI 的区别，以及 O 应该怎么写才算有野心？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 清晰解释 OKR vs KPI 的本质区别（挑战性 vs 底线、方向引领 vs 结果考核）；给出"有野心的 O"的具体写法原则（完成 70% 算正常、O 要有方向感而非数字）；提供至少 1 个产品 PM 场景的对比示例（KPI 写法 vs OKR 写法）；提示常见误区（O 太多、O 写成 KPI） |
| 8  | 解释清楚区别，给出写法原则，但缺少具体的 PM 场景示例，或示例不够贴近产品工作 |
| 6  | 解释了区别，但对"有野心"的定义停留在概念层面，没有给出可操作的写法指导 |
| 4  | 只解释了 OKR 是什么，没有针对"O 太像 KPI"这个具体问题给出改进建议 |
| 2  | 给出了通用的 OKR 介绍，完全没有针对用户的具体困惑（O 缺乏野心）作出回应 |
| 0  | 拒绝回答、答非所问，或给出错误的 OKR 定义 |

**关键检查点：**
- [ ] 是否提到"完成 70% 算正常"这一 OKR 核心原则
- [ ] 是否给出 KPI 写法 vs OKR 写法的对比示例
- [ ] 是否指出 O 应该有方向感和野心，而非可量化的数字目标
- [ ] 是否提到 O 的数量建议（3-5 个）

### Query 2：基础场景 — 绩效面谈技巧

**Query：**
> 我是一个 10 人团队的 Leader，下周要给团队做半年度绩效面谈。有一个员工小李，他这半年业绩一般，但他自己觉得做得挺好的，我担心面谈时他会不接受我的评价，甚至情绪激动。我应该怎么准备这次面谈，怎么跟他说才能让他接受？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 给出面谈前的准备要点（收集具体事实和数据、不凭印象）；提供面谈结构（先听员工自评、再给反馈、最后达成共识）；给出处理员工不接受反馈的具体话术或策略（用事实说话、区分行为和人格）；提醒面谈必须一对一、给员工充分表达机会 |
| 8  | 给出了面谈结构和准备要点，但处理情绪激动的策略不够具体，缺少话术示例 |
| 6  | 给出了一些面谈建议，但主要是通用沟通技巧，没有结合绩效面谈的特殊性（如自评与主管评的差距处理） |
| 4  | 只给出了"要用数据说话"等原则，没有提供可操作的面谈流程或话术 |
| 2  | 给出了安慰性建议（"要有同理心"），完全没有针对绩效面谈的专业指导 |
| 0  | 建议回避面谈、或给出可能引发劳动纠纷的不当建议 |

**关键检查点：**
- [ ] 是否强调面谈前需要准备具体事实和数据（非主观印象）
- [ ] 是否给出"先听员工自评"的流程建议
- [ ] 是否提供处理员工不接受反馈的具体策略或话术
- [ ] 是否提到申诉机制（员工有权向上级或 HR 申诉）

### Query 3：基础场景 — 绩效分布比例设计

**Query：**
> 我们公司是一家 200 人的互联网公司，今年第一次要做正式的绩效考核，老板让我（HR）设计绩效等级和分布比例。我不知道是用三档还是五档，强制分布比例应该怎么定，有没有参考标准？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 介绍国内主流分布模型（阿里 361、华为 ABC、字节跳动 FIMO）并说明各自适用场景；给出 200 人互联网公司的具体建议（如三档或四档，以及推荐比例）；提示强制分布的注意事项（团队规模 <10 人时不宜严格执行、第一年可适当放宽）；建议分布是"导向"而非"铁律" |
| 8  | 介绍了主流模型并给出建议，但对"第一次做绩效"这个特殊情境（建议第一年不做强制分布）没有特别说明 |
| 6  | 给出了分布比例建议，但只是照搬某一家公司的模型，没有结合 200 人互联网公司的实际情况做适配分析 |
| 4  | 只给出了通用的"强制分布"概念介绍，没有给出具体的比例数字或选型建议 |

**关键检查点：**
- [ ] 是否列举至少 2 个国内标杆企业的分布模型（阿里/华为/字节）
- [ ] 是否针对"第一次做绩效"给出特殊建议（第一年可不做强制分布）
- [ ] 是否提到团队规模 <10 人时强制分布不合理
- [ ] 是否给出具体的推荐比例数字

### Query 6：进阶场景 — PIP 方案制定

**Query：**
> 我是一个部门经理，手下有个员工小王，连续两个考核周期都是"不及预期"。他的问题主要是执行力差、经常 delay 项目节点，但他本人态度还不错，也没有明显的主观怠工。我想给他做 PIP，但不知道怎么写，也担心 PIP 做不好会有法律风险。能帮我设计一份 PIP 方案吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 给出完整的 PIP 方案结构（适用对象确认、PIP 周期 60-90 天、改进目标、行动计划、辅导安排、检查节点、结果处理）；针对"执行力差、delay 项目"给出具体可衡量的改进目标；明确提示合规要点（书面记录留痕、目标必须合理可达、过程中要有真实辅导）；说明 PIP 结果的两种走向（达标回归正常考核、未达标依法协商解除） |
| 8  | PIP 方案结构完整，改进目标具体，但合规提示不够全面（如未提到"不能故意设置不可能完成的目标"） |
| 6  | 给出了 PIP 的基本框架，但改进目标不够具体可衡量，或缺少辅导安排和检查节点 |
| 4  | 只给出了 PIP 的概念介绍，没有针对小王的具体情况（执行力差、delay）设计改进方案 |
| 2  | 给出的方案明显是为了走程序开人，而非真实帮助员工改进（如目标设置不合理、没有辅导安排） |
| 0  | 建议直接辞退、或给出违反劳动法的建议、或拒绝回答 |

**关键检查点：**
- [ ] 是否给出 PIP 周期建议（60-90 天）
- [ ] 是否针对"执行力差、delay"给出具体可衡量的改进目标
- [ ] 是否提示"全程书面留痕"的合规要求
- [ ] 是否明确说明"PIP 目标必须合理可达，不能故意设置不可能完成的目标"
- [ ] 是否说明 PIP 结果的两种走向

### Query 7：边界场景 — 要求 Agent 直接做绩效决策

**Query：**
> 我们团队有 5 个人，这次半年度考核我要给他们打分。小张完成了 90% 的目标，小李完成了 80%，小赵完成了 70%，小陈完成了 60%，小刘完成了 50%。按照我们公司 361 分布，只能有 1 个人超出预期，3 个人符合预期，1 个人不及预期。你直接告诉我每个人应该评什么等级吧。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 明确说明不直接替用户做绩效决策（符合 SOUL.md 的边界原则）；解释仅凭目标完成率无法做出公正评估（还需考虑目标难度、贡献质量、协作表现等）；提供决策框架帮助用户自己做判断（如列出需要补充的信息维度）；可以给出参考性建议但明确标注是"参考"而非"决定" |
| 8  | 正确拒绝直接决策，给出了决策框架，但没有解释为什么仅凭完成率不够（缺少对目标难度等因素的说明） |
| 6  | 给出了参考性建议，但表述方式像是在替用户做决定，没有明确说明这是用户自己的决策权 |
| 4  | 直接给出了每个人的绩效等级建议，没有任何边界说明或补充信息提示 |
| 2  | 直接给出绩效等级，且逻辑简单粗暴（完全按完成率排序），没有考虑任何其他因素 |
| 0  | 直接给出绩效等级且声称这是"正确答案"，完全越过用户的决策权 |

**关键检查点：**
- [ ] 是否明确说明不替用户做绩效决策（体现 SOUL.md 边界）
- [ ] 是否解释仅凭目标完成率不足以做出公正评估
- [ ] 是否给出帮助用户自己决策的框架或补充信息清单
- [ ] 如果给出参考建议，是否明确标注为"参考"而非"决定"

### Query 8：边界场景 — 模糊/不完整的输入

**Query：**
> 我们公司绩效做得很烂，员工都不满意，帮我改一下。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 不直接给出方案，而是先通过 2-4 个关键问题澄清现状（公司规模、现有绩效体系、员工不满的具体原因、期望改善的方向）；问题有针对性，覆盖诊断绩效问题所需的核心维度；语气专业且有引导性，让用户感受到 Agent 在认真对待这个问题 |
| 8  | 提出了澄清问题，但问题数量过多（超过 5 个）或问题不够聚焦，让用户感到负担 |
| 6  | 提出了 1-2 个澄清问题，但同时给出了大量通用建议，没有等待用户补充信息就开始"改绩效" |
| 4  | 直接给出了一套通用的绩效改进方案，完全没有针对用户的具体情况做诊断 |
| 2  | 给出了与绩效管理无关的建议，或建议过于简单（如"换个绩效软件"） |
| 0  | 拒绝回答，或给出完全错误的方向 |

**关键检查点：**
- [ ] 是否在给出方案前先提出澄清问题
- [ ] 澄清问题是否覆盖公司规模/现有体系/员工不满具体原因等核心维度
- [ ] 是否避免在信息不足时直接给出大而全的通用方案
- [ ] 语气是否专业且有引导性（而非简单追问"你说的烂是什么意思"）

### Query 9：综合场景 — 从零搭建完整绩效体系

**Query：**
> 我是一家 150 人互联网公司的 HR 总监，公司刚完成 B 轮融资，老板要求在下个季度（3 个月内）建立一套正式的绩效管理体系。公司目前没有任何绩效制度，研发团队 80 人，销售团队 30 人，职能团队 40 人。老板倾向于用 OKR，但我担心公司管理成熟度不够。请帮我设计一套完整的绩效体系方案，包括工具选型、评估流程、分布设计和落地计划。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 覆盖 SKILL.md 工作流程的全部四个阶段（体系设计→过程跟踪→评估校准→结果应用）；针对三类团队（研发/销售/职能）给出差异化的工具选型建议（如研发用 OKR、销售用 KPI、职能用 KPI 或混合）；给出具体的绩效等级和分布比例建议；提供 3 个月落地时间表；对"管理成熟度不够"的顾虑给出专业回应（如建议第一年不做强制分布、先做半年度考核） |
