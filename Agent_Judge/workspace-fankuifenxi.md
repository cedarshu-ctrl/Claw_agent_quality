# 反馈分析师（workspace-vakm3yibyqrfxh5f）— LLM-AS-Judge 评测标准

## Agent 背景

反馈分析师是一位"用户声音翻译官"，核心能力是把来自 App Store、客服工单、社交媒体、NPS 调研等多渠道的碎片化用户反馈，转化为可执行的产品改进洞察。它不只是做分类统计，而是透过用户原话找到真实需求，区分"吵得最凶"和"真正重要"，并推动反馈闭环落地。

其标志性交付物包括：结构化的反馈周报（含 Top 问题、严重程度分布、情感得分）、根因分析报告、产品优先级建议，以及基于 FeedbackAnalyzer 数据类的技术分析代码。Agent 的核心信条是"单条反馈是故事，多条反馈才是数据"，每个洞察必须附上样本数和置信度。

---

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | 产品经理、用户研究员、增长运营，所在团队有一定用户规模（日活万级以上），需要定期处理多渠道用户反馈 |
| 核心需求 | 从每天涌入的大量反馈中快速识别真正重要的问题，避免被噪音带偏，给出有数据支撑的产品决策依据 |
| 使用频率 | 每周固定（生成周报）+ 按需（某个问题突然爆发时紧急分析），典型触发时机是周一晨会前或版本迭代规划前 |
| 专业水平 | 中级——熟悉产品工作流程，但缺乏系统化的反馈分析方法论；期望 Agent 给出结构化输出而非泛泛建议 |

---

## 核心使用场景

1. **周报生成**：每周一产品会议前，需要把上周所有渠道的反馈汇总成一份可以直接汇报的周报，包含 Top 问题、趋势变化和紧急事项。
2. **需求真实性验证**：收到某个功能请求后，需要判断这是个别用户的偏好还是普遍痛点，以及用户真正想解决的问题是什么。
3. **版本迭代优先级排序**：下个版本有多个待修复问题，需要根据反馈数据决定哪些先做、哪些可以推后。
4. **负面舆情应急分析**：某功能上线后 App Store 评分骤降，需要快速定位是哪类用户在抱怨什么，以及问题的严重程度。
5. **反馈闭环追踪**：某个被用户反复提及的问题已经修复上线，需要确认用户是否感知到改进，并设计通知方案。

## 评测 Query 与打分标准

### Query 1：基础场景 — 反馈分类与标注

**Query：**
> 我们上周从 App Store、客服工单和微博收到了大概 200 条反馈，我把原文整理成了一个列表发给你。你能帮我把这些反馈分类、打标签、判断严重程度吗？我需要知道哪些是 Bug、哪些是功能请求、哪些是体验问题，以及哪些是紧急的。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按 Category（Bug/功能请求/体验问题/性能问题/正向反馈）和 Severity（Critical/High/Medium/Low）双维度分类；保留用户原话作为代表性引用；给出每类的数量和占比；Critical 和 High 问题单独列出并说明判断依据；输出格式可直接用于后续分析 |
| 8  | 分类和严重程度判断准确，但 Severity 判断标准未说明，或正向反馈未单独处理，或缺少数量统计 |
| 6  | 完成了基本分类，但只用了 2-3 个类别（如只分"问题"和"建议"），未按 SKILL.md 的五类体系；或严重程度只有笼统的"重要/不重要" |
| 4  | 对反馈做了简单归纳，但没有结构化分类，只是把相似反馈堆在一起，没有标签和严重程度 |
| 2  | 只是复述了用户反馈内容，没有做任何分类分析；或把所有反馈都标为同一严重程度 |
| 0  | 拒绝处理（以"没有具体数据"为由不给任何分析框架）；或输出与分类任务完全无关 |

**关键检查点：**
- [ ] 是否使用了 Category 五分类体系（Bug/功能请求/体验问题/性能问题/正向反馈）
- [ ] 是否使用了 Severity 四级体系（Critical/High/Medium/Low）并说明判断标准
- [ ] 是否保留了用户原话作为每类的代表性引用
- [ ] 是否给出了各类别的数量和百分比
- [ ] Critical 问题是否被单独标注并给出处理建议

### Query 2：基础场景 — 生成反馈周报

**Query：**
> 我是一个 SaaS 产品的 PM，每周一要给老板汇报上周的用户反馈情况。上周我们收到了 180 条反馈，主要集中在"导出功能报错"（38 条）、"加载速度慢"（29 条）、"希望支持多人协作"（22 条）、"界面不够直观"（18 条）、"登录经常掉线"（15 条），另外有 31 条是正向反馈。帮我生成一份周报。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 包含：本周总量和环比趋势（如有）；Top 5 问题排名及各自数量和占比；严重程度判断（"导出报错"和"登录掉线"应被标为 Critical/High）；情感分布（正向反馈占比 17%）；紧急行动项（Critical 问题需本周处理）；格式清晰可直接用于汇报 |
| 8  | 包含 Top 问题排名和数量，有严重程度判断，但缺少情感分布统计，或行动建议过于笼统（"尽快修复"而非具体时间节点） |
| 6  | 列出了 Top 问题，但没有严重程度分级，也没有区分 Bug 类（需紧急处理）和功能请求类（可排期），行动建议缺失 |
| 4  | 只是把数字重新排列了一遍，没有分析和洞察，没有指出哪些问题更紧急 |
| 2  | 输出格式混乱，或把功能请求（多人协作）和 Bug（导出报错）混为一谈，没有区分处理优先级 |
| 0  | 拒绝生成周报；或输出内容与给定数据完全不符 |

**关键检查点：**
- [ ] 是否识别出"导出功能报错"和"登录掉线"为高优先级 Bug（而非普通反馈）
- [ ] 是否给出了正向反馈的占比（31/180 ≈ 17%）
- [ ] 是否区分了 Bug 类和功能请求类，并给出不同的处理建议
- [ ] 是否包含可直接汇报的结构（总量 → Top 问题 → 紧急事项 → 行动建议）
- [ ] 行动建议是否具体（涉及时间节点或责任方）

### Query 3：进阶场景 — 需求真实性翻译

**Query：**
> 最近两周我们收到了 35 条用户反馈，都在说"能不能加个导出 PDF 的功能"。我们开发资源很紧张，这个功能要做的话大概需要两周。我想知道这 35 个用户真正想要的是什么，以及我们是不是一定要做 PDF 导出？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 进行需求翻译：分析"导出 PDF"背后的真实场景（分享给不用该产品的人、存档、打印汇报等）；提出替代方案（分享链接、截图优化、邮件发送）并对比开发成本；给出数据支撑的建议（35 条是否足够支撑两周开发投入，需结合用户层级分布）；明确说明"用户说要 A 但真正需要 B"的分析逻辑 |
| 8  | 做了需求翻译并提出了替代方案，但没有结合用户层级（付费用户 vs 免费用户）分析，或没有给出开发优先级建议 |
| 6  | 指出了用户可能有更深层需求，但只是泛泛而谈，没有具体分析场景，也没有提出替代方案 |
| 4  | 直接建议"做 PDF 导出"或"不做 PDF 导出"，没有做需求翻译，只是基于数量做判断 |
| 2  | 回答与需求分析无关，或只是重复了用户的原始请求 |
| 0  | 拒绝分析；或建议"因为一个用户吼得最凶就改排期"（违反 SOUL.md 边界规则） |

**关键检查点：**
- [ ] 是否识别出"导出 PDF"背后的真实使用场景（至少列举 2 个）
- [ ] 是否提出了至少 1 个替代方案并分析其开发成本对比
- [ ] 是否提到需要结合用户层级（付费/免费）来判断优先级
- [ ] 是否给出了明确的行动建议（做/不做/做替代方案）及理由
- [ ] 是否体现了"单条反馈是故事，多条反馈才是数据"的分析原则

### Query 4：进阶场景 — 用户分层交叉分析

**Query：**
> 我们的产品有免费用户、Pro 用户和企业用户三类。最近我发现免费用户的差评特别多，但 Pro 用户和企业用户反馈还好。老板说要优先解决免费用户的问题，因为数量最多。我觉得这个判断可能有问题，但不知道怎么反驳。你能帮我分析一下吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 提供用户分层分析框架：说明不同层级用户的反馈权重应该不同（付费用户流失成本更高）；区分"反馈数量多"和"业务影响大"；给出具体的分析建议（如计算各层级的情感得分、流失率关联、LTV 影响）；提供可以用来说服老板的数据框架；引用 sentiment_by_tier() 方法的分析思路 |
| 8  | 指出了用户分层的重要性，给出了分析框架，但没有提供具体的数据指标或说服老板的话术 |
| 6  | 认同了"不应该只看数量"的判断，但分析停留在定性层面，没有给出量化分析方法 |
| 4  | 只是支持了用户的直觉，没有提供任何分析框架或数据支撑 |
| 2  | 支持老板的判断（"免费用户多所以要优先处理"），没有做分层分析 |
| 0  | 拒绝分析；或给出与用户分层完全无关的建议 |

**关键检查点：**
- [ ] 是否明确指出"反馈数量"和"业务影响"是两个不同维度
- [ ] 是否提到付费用户（Pro/企业）的流失成本高于免费用户
- [ ] 是否给出了具体的分析指标（如各层级情感得分、流失率、LTV）
- [ ] 是否提供了可以用数据说服老板的具体论据框架
- [ ] 是否建议同时关注两类用户而非非此即彼

### Query 5：进阶场景 — 根因分析与洞察报告

**Query：**
> 我们的 App Store 评分上个月从 4.5 降到了 4.1，老板很着急。我把最近 30 天的 1 星和 2 星评价整理了一下，发现主要集中在三类：第一类是"更新后闪退"（52 条），第二类是"新版界面太难用了"（41 条），第三类是"客服不回复"（28 条）。帮我做一个根因分析，找出最需要优先解决的问题。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 对三类问题分别做根因分析：闪退（技术 Bug，Critical，直接影响可用性）、界面（体验问题，High，需要用户研究验证）、客服（流程问题，High，影响信任度）；给出优先级排序及理由；量化每类问题对评分的影响估算；给出每类问题的具体行动建议（时间节点、责任方）；指出"更新后闪退"是最紧急的因为它是 Critical Bug |
| 8  | 完成了根因分析和优先级排序，但缺少对评分影响的量化估算，或行动建议缺少时间节点 |
| 6  | 指出了三类问题的性质差异，但优先级排序理由不充分，或把三类问题都标为同等优先级 |
| 4  | 只是重复了三类问题的描述，没有做根因分析，没有优先级判断 |
| 2  | 优先级排序错误（如把"客服不回复"排在"闪退"前面），或根因分析与实际问题不符 |
| 0  | 拒绝分析；或建议"因为客服问题条数少就不处理" |

**关键检查点：**
- [ ] 是否将"更新后闪退"识别为 Critical 级别（最高优先级）
- [ ] 是否对三类问题分别给出了根因假设（技术/设计/流程）
- [ ] 是否给出了优先级排序并说明理由（不只是按数量排）
- [ ] 是否给出了每类问题的具体行动建议
- [ ] 是否提到了评分恢复的预期时间或条件

### Query 6：边界场景 — 单条反馈驱动排期

**Query：**
> 我们有个大客户，他们的 CEO 亲自发邮件说希望我们下个版本加上"批量导入 Excel"功能，不然他们可能考虑换竞品。这个功能我们之前没有收到过其他用户的反馈。你觉得我们应该立刻把这个功能加到下个版本吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 明确指出不应该因为单个用户（即使是大客户 CEO）就立刻改排期；建议先验证：查看其他企业用户是否有类似需求、评估该客户的 LTV 和流失风险、在反馈库中搜索相关关键词；给出决策框架（商业价值 × 用户覆盖面 × 开发成本）；同时建议与该客户沟通时间预期，避免直接承诺 |
| 8  | 建议不立刻改排期，给出了验证步骤，但缺少具体的决策框架或与客户沟通的建议 |
| 6  | 建议先调研其他用户，但态度模糊（"可以考虑做"），没有明确说明单条反馈不足以驱动排期的原则 |
| 4  | 建议立刻做这个功能（因为是大客户），违反了"不因为一个用户吼得最凶就改排期"的原则 |
| 2  | 完全忽视商业风险，建议拒绝大客户需求而不给任何替代方案 |
| 0  | 直接说"大客户说什么就做什么"；或拒绝给出任何建议 |

**关键检查点：**
- [ ] 是否明确引用"单条反馈是故事，多条反馈才是数据"的原则
- [ ] 是否建议先验证其他用户是否有类似需求
- [ ] 是否给出了商业价值评估框架（不只是技术可行性）
- [ ] 是否建议与客户沟通时间预期而非直接承诺
- [ ] 是否在拒绝立刻排期的同时给出了建设性的替代方案

### Query 7：边界场景 — 模糊/不完整输入

**Query：**
> 我们最近用户反馈不太好，你帮我分析一下。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 识别出输入不完整，主动询问关键缺失信息（反馈来源、时间范围、数量、具体内容）；同时提供一个分析框架模板，让用户知道提供什么信息后可以得到什么样的分析；语气专业但不冷漠，体现共情（"反馈不好"说明有压力） |
| 8  | 询问了关键缺失信息，但没有提供分析框架模板，用户不清楚提供信息后能得到什么 |
| 6  | 给出了通用的反馈分析建议，但没有针对用户的具体情况，也没有询问缺失信息 |
| 4  | 给出了大量泛泛的反馈分析方法论，但完全没有针对用户的实际需求 |
| 2  | 直接开始"分析"，但因为没有数据只能输出空洞的内容 |
| 0  | 拒绝回应；或要求用户"先去收集数据再来找我" |

**关键检查点：**
- [ ] 是否识别出输入缺少具体数据（反馈内容、数量、来源）
- [ ] 是否主动询问了至少 3 个关键缺失信息
- [ ] 是否提供了分析框架模板（告知用户提供什么信息可以得到什么输出）
- [ ] 是否体现了共情（而非冷漠地要求补充信息）

### Query 8：综合场景 — 端到端反馈分析与产品建议

**Query：**
> 我是一个 B2B SaaS 产品的 PM，我们产品有 1200 个付费客户（其中 80 个企业客户，1120 个 Pro 用户）。上个月我们做了一次大版本更新，更新后收到了 320 条反馈，比平时多了 3 倍。我把反馈整理了一下：Bug 类 89 条（其中 12 条是企业客户反馈的数据同步失败）、体验问题 76 条（主要是新导航栏不习惯）、功能请求 94 条（最多的是"希望支持 API 集成"，有 38 条）、正向反馈 61 条。我需要你帮我：1）判断哪些问题最紧急；2）分析"API 集成"需求的真实性；3）给出下个版本的优先级建议；4）设计一个反馈闭环方案。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 完整覆盖四个子任务：①紧急度判断：企业客户数据同步失败为 Critical（12 条企业客户 Bug > 89 条普通 Bug），需 24-48 小时内响应；②API 集成需求翻译：38 条中分析真实场景（自动化工作流、与现有系统集成），建议先做 Webhook 或轻量集成验证需求；③优先级建议：Critical Bug → 体验问题（影响留存）→ API 集成（影响扩展）→ 其他功能请求；④闭环方案：企业客户 1 对 1 通知、Pro 用户邮件/应用内通知、更新日志说明改进来源于用户反馈 |
| 8  | 覆盖了四个子任务，但某一项分析深度不足（如优先级建议缺少理由，或闭环方案只提到"通知用户"而无具体方式） |
| 6  | 覆盖了 2-3 个子任务，遗漏了 1 个；或四个都有但都停留在表面，缺少数据支撑和具体建议 |
| 4  | 只回答了 1-2 个子任务，其余被忽略；或把企业客户 Bug 和普通 Bug 同等对待 |
| 2  | 回答混乱，四个子任务没有清晰区分；或优先级判断严重错误（如把功能请求排在 Critical Bug 前面） |
| 0  | 拒绝回答；或输出内容与四个子任务完全无关 |

**关键检查点：**
- [ ] 是否将"企业客户数据同步失败"识别为最高优先级（Critical，高于数量更多的普通 Bug）
- [ ] 是否对"API 集成"需求做了翻译（分析真实使用场景而非直接建议做/不做）
- [ ] 是否给出了四个维度的完整优先级排序并附理由
