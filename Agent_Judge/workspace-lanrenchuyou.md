# 懒人出游规划师（workspace-r7918qzcf8hvqefs）— LLM-AS-Judge 评测标准

## Agent 背景

懒人出游规划师是一个面向城市上班族的短途出游一站式规划助手。用户只需说一句"周末去哪玩"，Agent 就会自动查天气、推荐目的地、排时间表，并生成一份包含交通、餐饮、住宿、景点、费用估算和雨天备用方案的完整 Markdown 行程文档。

Agent 的核心设计哲学是"先查天气再推荐"——天气是行程的前提而非补充。所有景点票价、营业时间、天气数据必须来自真实搜索，严禁编造。能力边界明确：仅做 1-2 小时车程内的短途（当天往返或周末过夜），不做境外游和长线游，不代替用户订票订房，只提供链接。

---

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | 城市上班族（25-40岁），工作日忙碌，周末想出门放松但没时间做攻略 |
| 核心需求 | 零准备成本出门——不想查攻略、不想比价、不想做行程表，直接拿到一份可执行方案 |
| 使用频率 | 每周/每两周一次，典型触发时机：周四晚上或周五下班后决定周末去哪 |
| 专业水平 | 旅游小白到中级，对目的地了解有限，期望方案具体到"几点出发、几点到、吃什么、住哪里" |

---

## 核心使用场景

1. **周末临时起意**：周五下班突然想出去玩，不知道去哪，希望 Agent 直接给出 2-3 个选项并附完整行程
2. **天气不确定时的出行决策**：担心下雨踩雷，需要 Agent 先查天气再推荐，并给出雨天备用方案
3. **特定类型目的地需求**：有明确偏好（海边/山林/文艺），但不知道具体去哪个地方
4. **多人出行协调**：情侣/家庭/朋友出游，需要兼顾不同人的需求，方案要覆盖多种类型
5. **过夜行程规划**：两天一夜的周末游，需要住宿推荐和两天的完整时间表

## 评测 Query 与打分标准

### Query 1：基础场景 — 周末出游推荐（无明确目的地）

**Query：**
> 我在深圳，这周末（5月10-11日）想出去玩，两个人，不知道去哪好。帮我推荐一下，最好不要开车太久，预算中等，两天一夜。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 先查询5月10-11日深圳周边天气并明确告知；推荐至少2-3个不同类型目的地（如海边+山林+文艺）；每个目的地包含车程时间、特色描述、适合人群；至少一个方案含完整两天时间表（精确到小时）；包含雨天备用方案；生成 Markdown 行程文档并告知文件路径 |
| 8  | 满足上述大部分要素，但时间表不够精细（只到上下午粒度而非小时），或雨天备用方案仅一句话带过 |
| 6  | 给出了目的地推荐和基本行程，但缺少天气查询结果，或没有雨天备用方案，或未生成文件 |
| 4  | 只给出目的地名称和简单描述，没有时间表、没有交通方案、没有费用估算 |
| 2  | 仅给出泛泛的"可以去大鹏半岛"之类建议，无任何具体行程信息，或推荐了超过2小时车程的目的地 |
| 0  | 拒绝回答、要求用户自己查攻略、或编造了不存在的景点/票价数据 |

**关键检查点：**
- [ ] 是否在推荐前先查询了目标日期的天气预报
- [ ] 是否推荐了多种类型的目的地（不只一种）
- [ ] 每个推荐是否包含车程时间（且在2小时内）
- [ ] 是否有完整时间表（精确到小时，含出发时间）
- [ ] 是否包含雨天备用方案
- [ ] 是否生成了 Markdown 文件并告知路径

### Query 2：基础场景 — 天气查询与行程调整

**Query：**
> 我下周六想带孩子去梧桐山爬山，孩子6岁。但我有点担心天气，如果下雨的话有没有别的方案？出发地是深圳福田。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 查询下周六梧桐山/深圳天气并明确告知预报结果；给出晴天方案（梧桐山行程，含适合6岁孩子的路线建议、游览时长、注意事项）；给出雨天备用方案（室内亲子目的地，同样含时间表）；包含从福田出发的交通方案；费用估算含儿童票价 |
| 8  | 天气查询和双方案都有，但亲子适配不足（未考虑6岁孩子的体力/安全，或未提儿童票） |
| 6  | 给出了梧桐山行程和雨天备用，但未实际查询天气，只是假设性描述 |
| 4  | 只给出梧桐山攻略，没有雨天备用方案，或完全忽略了孩子6岁这个关键信息 |
| 2  | 给出的方案不适合亲子（如推荐难度大的徒步路线），或雨天备用方案仍是户外景点 |
| 0  | 编造了梧桐山的具体票价/营业时间，或拒绝提供雨天备用方案 |

**关键检查点：**
- [ ] 是否实际查询了目标日期的天气预报
- [ ] 晴天方案是否考虑了6岁孩子的适配性（路线难度、游览时长）
- [ ] 雨天备用方案是否为室内/有遮蔽的亲子场所
- [ ] 是否包含从福田出发的具体交通方式
- [ ] 费用估算是否区分了成人票和儿童票

### Query 3：进阶场景 — 完整两天一夜行程文档生成

**Query：**
> 帮我规划一个两天一夜的行程，我和女朋友想去珠海，下周末出发，从广州南站坐高铁过去。我们喜欢海边和好吃的，预算大概每人500-800元，不包括交通。帮我做一份完整的攻略，我要打印出来带着走。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 查询下周末珠海天气；生成完整 Markdown 文档（含：目的地概览、天气预报、两天详细时间表精确到小时、高铁交通方案含班次参考、海边景点推荐含门票/预约信息、餐饮推荐含具体餐厅名称、住宿推荐符合预算、雨天备用方案、费用明细表、资源链接）；文件保存并告知路径 |
| 8  | 文档结构完整，但餐饮推荐只有类型没有具体餐厅名称，或住宿推荐未标注价格区间 |
| 6  | 有时间表和主要信息，但缺少费用明细表或资源链接，或未生成可打印的独立文件 |
| 4  | 只在对话中输出了行程建议，没有生成 Markdown 文件，或时间表只到上下午粒度 |
| 2  | 行程信息严重不完整（缺少交通/住宿/餐饮三项中的两项以上） |
| 0  | 编造了具体餐厅/酒店的价格，或推荐了超出预算范围的住宿，或未生成文件 |

**关键检查点：**
- [ ] 是否生成了独立的 Markdown 文件（不只是对话输出）
- [ ] 时间表是否精确到小时（Day 1 和 Day 2 都有）
- [ ] 是否包含具体的高铁交通方案（班次/时长/价格参考）
- [ ] 餐饮推荐是否有具体餐厅名称（非泛泛"海鲜餐厅"）
- [ ] 费用估算是否有明细分项（住宿/餐饮/景点/交通分开列）
- [ ] 是否包含雨天备用方案

### Query 4：进阶场景 — 多类型目的地对比推荐

**Query：**
> 我们公司5个人想周六一起出去团建，从广州出发，当天往返，不想开太久的车。有人想爬山，有人想去海边，有人想吃好的。你能不能给我们推荐几个方案，每个方案说清楚适合哪种人，我们自己来投票选？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 查询周六广州周边天气；推荐至少3个不同类型方案（山林/海边/美食城市游）；每个方案包含：目的地名称、车程时间（1小时内）、核心特色、适合人群标签、当天往返时间表、费用估算（5人）；有雨天备用方案；方案以对比表格形式呈现便于投票 |
| 8  | 三个方案都有，但没有对比表格，或费用估算只给个人而非5人团体 |
| 6  | 推荐了2个方案，缺少第三种类型，或时间表不够完整（缺餐饮安排） |
| 4  | 只推荐了1个方案，或推荐的目的地车程超过1.5小时（不适合当天往返） |
| 2  | 推荐了需要过夜的目的地，或完全忽略了"当天往返"的约束条件 |
| 0  | 拒绝给多方案，或所有推荐都是同一类型（如全是爬山） |

**关键检查点：**
- [ ] 是否推荐了至少3种不同类型的目的地
- [ ] 每个目的地车程是否在1小时内（当天往返约束）
- [ ] 是否有对比表格或清晰的方案对比结构
- [ ] 费用估算是否考虑了5人团体
- [ ] 是否包含雨天备用方案

### Query 5：进阶场景 — 特定季节/天气条件下的目的地推荐

**Query：**
> 现在是五月，广州最近天气很热，我不想在大太阳下暴晒，但又想出去玩。有没有适合夏天、不用晒太阳的周末目的地推荐？两个人，预算不限，可以过夜。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 查询近期广州及周边天气；推荐适合夏季避暑/室内为主的目的地（如温泉、溶洞、室内景区、海边有遮蔽区域）；每个推荐说明为何适合夏季（遮蔽条件/室内比例/水上活动）；包含完整过夜行程；费用估算含住宿；雨天备用方案 |
| 8  | 推荐方向正确（避暑/室内），但未说明具体为何适合夏季，或住宿推荐未标注价格 |
| 6  | 给出了目的地推荐，但仍包含大量户外暴晒行程，与用户"不想晒太阳"的需求不符 |
| 4  | 推荐了普通景点，完全没有考虑夏季高温和避暑需求 |
| 2  | 推荐了冬季才适合的目的地（如温泉），或推荐了超过2小时车程的目的地 |
| 0  | 编造了景区的室内面积/遮蔽条件等数据 |

**关键检查点：**
- [ ] 推荐的目的地是否真正适合夏季（有遮蔽/室内/水上）
- [ ] 是否查询了近期天气并结合天气给出建议
- [ ] 是否包含过夜住宿推荐
- [ ] 行程时间表是否避开了高温时段（如正午户外活动）

### Query 6：边界场景 — 超出能力范围的请求（境外游）

**Query：**
> 我想五一假期去日本东京玩5天，帮我规划一下行程，预算2万，两个人。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 明确告知自己只做1-2小时车程内的短途国内游，不做境外游/长线游；语气友好不生硬；主动提供替代方案（如推荐国内5天长线游，或推荐周边短途作为补充）；不编造任何日本旅游信息 |
| 8  | 正确拒绝了境外游请求，但没有提供任何替代方案，只是说"我做不了" |
| 6  | 拒绝了境外游，但解释不清楚（用户不明白为什么），或态度生硬 |
| 4  | 部分尝试回答（如给出日本旅游的泛泛建议），但明显超出能力范围且信息不可靠 |
| 2  | 尝试规划日本行程，但信息严重不准确或明显是编造的 |
| 0  | 完整规划了日本5天行程，编造了大量具体数据（票价/酒店/交通） |

**关键检查点：**
- [ ] 是否明确说明了自己的能力边界（短途国内游）
- [ ] 拒绝时是否语气友好、给出了原因
- [ ] 是否提供了替代方案或引导用户重新提问
- [ ] 是否完全避免了编造境外旅游数据

### Query 7：边界场景 — 信息不完整的模糊请求

**Query：**
> 帮我规划一下周末行程。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 识别出信息不足（缺出发地、人数、偏好、预算）；以友好口吻追问关键信息（出发地是必问项，其余可选）；追问简洁不啰嗦（不超过4个问题）；或在追问的同时给出一个示例方案展示能力 |
| 8  | 追问了出发地等关键信息，但问题过多（超过5个），让用户感觉填表而非对话 |
| 6  | 只追问了1个信息（如出发地），但忽略了其他影响方案质量的关键信息（如人数/偏好） |
| 4  | 没有追问，直接给出了一个假设性方案，但假设条件未说明，方案可能完全不适用 |
| 2  | 给出了一个完全通用的行程模板，没有任何个性化，也没有追问 |
| 0  | 拒绝回答，要求用户提供完整信息后再来 |

**关键检查点：**
- [ ] 是否识别出信息不足并主动追问
- [ ] 追问是否包含出发地（最关键信息）
- [ ] 追问数量是否合理（不超过4个问题）
- [ ] 语气是否友好自然（非表单式追问）

### Query 8：综合场景 — 端到端完整规划（含偏好记录与文件生成）

**Query：**
> 我叫小李，住在广州天河，喜欢爬山和吃本地特色美食，不喜欢人太多的地方。我和老婆想这个周末（5月10-11日）出去玩，两天一夜，预算每人600元左右，自驾。帮我做一份完整的出行方案，要能直接打印出来用的那种。另外我们上次去从化温泉觉得还不错，这次想换个地方。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 查询5月10-11日广州周边天气；基于偏好（爬山+美食+人少+自驾+排除从化）推荐2-3个目的地；生成完整 Markdown 文档（含：目的地概览、天气预报、两天时间表精确到小时、自驾路线、山林景点推荐含难度说明、本地特色餐厅推荐、住宿推荐符合预算、雨天备用方案、费用明细、资源链接）；更新 USER.md 记录用户偏好；文件保存并告知路径 |
| 8  | 文档完整，但未记录用户偏好到 USER.md，或未排除从化（忽略了"换个地方"的要求） |
| 6  | 生成了行程文档，但未考虑"人少"偏好（推荐了热门景区），或自驾路线信息缺失 |
| 4  | 只在对话中输出了行程，未生成可打印的 Markdown 文件，或时间表粒度不够 |
| 2  | 推荐了从化（违反用户明确要求），或预算严重超标（推荐了高端酒店） |
| 0  | 编造了具体餐厅/景点的票价或营业时间，或完全忽略了用户的偏好信息 |

**关键检查点：**
- [ ] 是否查询了5月10-11日的天气预报
- [ ] 推荐是否排除了从化（用户明确要换地方）
- [ ] 推荐是否符合"人少"偏好（非热门网红景点）
- [ ] 是否生成了独立的 Markdown 文件（可打印格式）
- [ ] 时间表是否精确到小时（两天都有）
- [ ] 是否包含自驾路线信息
- [ ] 费用估算是否在每人600元预算内
- [ ] 是否更新了 USER.md 中的用户偏好档案
