# 文件对比专家（workspace-kq2ucsh4f483ss0a）— LLM-AS-Judge 评测标准

## Agent 背景

文件对比专家是一个专注于多文档差异检测与整合的 AI 智能体，定位为"文件差异侦探"。其核心能力涵盖逐行/逐单元格精准对比、合同条款变更标注、Excel 数据核对、多版本方案迭代追踪，以及多人反馈意见的去重归类整合。

Agent 的工作风格类似细心高效的审计员——以结构化对比表格为主要输出形式，强调"30 秒内搞清楚改了啥"的交付效率。标志性交付物为 Markdown 差异报告（含执行摘要、详细变更表格、去重建议清单），并要求将报告保存为文件而非仅在对话中输出。

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | 办公人员、项目经理、法务/合同专员、产品经理、文档管理员 |
| 核心需求 | 快速定位两个或多个版本文件之间的差异，避免人工逐行比对的低效和漏检风险 |
| 使用频率 | 按需触发，典型时机：收到修改版合同/方案时、多人协作后需整合反馈时、数据核对时 |
| 专业水平 | 中级，熟悉文档管理流程，但不具备编程能力，期望 Agent 自动完成技术性对比操作并给出可读报告 |

## 核心使用场景

1. **合同版本对比**：法务或业务人员收到对方修改后的合同，需要快速确认哪些条款被改动，尤其关注金额、期限、责任条款等关键内容。
2. **方案迭代追踪**：产品或项目经理在多轮评审后，需要梳理方案从 V1 到 V3 的演进脉络，确认每轮改动是否落实了评审意见。
3. **Excel 数据核对**：运营或财务人员需要比对两份报表（如月初预算 vs 月末实际），找出数值差异行。
4. **多人反馈整合**：文档负责人收集了来自多位同事的修改意见，需要去重归类后整合到一个版本。
5. **代码/配置文件 Review**：开发或运维人员需要对比两个版本的配置文件或脚本，确认变更范围。

## 评测 Query 与打分标准

### Query 1：基础场景 — 合同条款差异对比

**Query：**
> 我是公司法务，刚收到供应商发来的合同修改版。我手上有原版（contract_v1.docx）和他们改过的版本（contract_v2.docx），两个文件都在桌面上。我需要知道他们到底改了哪些地方，特别是付款条款和违约责任那部分有没有动过。能帮我对比一下吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 先确认文件路径，然后读取两个文件并运行对比脚本；输出包含：执行摘要（一段话说明整体改动量）、结构化变更表格（逐条列出新增/删除/修改的条款，标注条款编号和变更类型）、对付款条款和违约责任的专项标注、明确提示"涉及重要条款建议人工核对"、将报告保存为文件并告知路径 |
| 8  | 完成对比并输出结构化报告，但未专项标注付款/违约条款，或未提示人工核对建议 |
| 6  | 输出了差异内容，但格式为纯文本流水账而非结构化表格，或未区分新增/删除/修改类型 |
| 4  | 仅描述了对比思路或列出部分差异，未完整覆盖两个文件的全部变更 |
| 2  | 转述了合同原文大段内容，违反"不转述完整原文"的边界规则，或仅给出通用建议未实际执行对比 |
| 0  | 拒绝执行对比、未读取文件直接编造差异内容、或完全答非所问 |

**关键检查点：**
- [ ] 是否先确认了文件路径（步骤 1：收集用户需求）
- [ ] 是否输出了包含新增/删除/修改分类的结构化变更表格
- [ ] 是否对付款条款和违约责任进行了专项标注
- [ ] 是否包含"建议人工核对"的提示（符合 Boundaries 规则）
- [ ] 是否将报告保存为文件并告知路径（SKILL.md 验收标准）

### Query 2：基础场景 — Excel 数据差异核对

**Query：**
> 我们运营部门每个月都要核对两份销售数据表，一份是系统导出的（sales_system.xlsx），一份是业务员手填的（sales_manual.xlsx）。这个月发现总数对不上，但我不知道是哪几行出了问题。两个表格格式一样，都是按订单号排的。帮我找出哪些行的数据不一致？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 确认文件路径和对比维度（以订单号为 key）；运行对比脚本；输出包含：执行摘要（不一致行数/总行数）、差异明细表格（订单号、字段名、系统值、手填值、差值）、去重建议（如重复订单号处理方式）、报告保存为文件 |
| 8  | 完成对比并输出差异明细，但未计算差值列，或未说明总体不一致比例 |
| 6  | 找出了部分不一致行，但未以表格形式呈现，或遗漏了某些字段的差异 |
| 4  | 仅说明了对比方法或给出示例，未实际读取文件执行对比 |
| 2  | 输出了大量原始数据内容而非差异摘要，或建议用户自己用 Excel 公式对比 |
| 0  | 声称无法处理 Excel 文件、拒绝执行、或编造不存在的差异数据 |

**关键检查点：**
- [ ] 是否确认了对比的 key 字段（订单号）
- [ ] 差异表格是否包含"系统值 vs 手填值"的并列对比列
- [ ] 是否输出了不一致行数占总行数的比例（执行摘要）
- [ ] 是否将报告保存为文件（write_to_file 要求）

### Query 3：进阶场景 — 多版本方案迭代追踪

**Query：**
> 我是产品经理，我们的 APP 改版方案已经经历了三轮评审，现在有 v1、v2、v3 三个版本的 PRD 文档。老板下周要开总结会，想知道这三个版本之间的演进脉络——每轮改了什么、为什么改。三个文件分别是 prd_v1.md、prd_v2.md、prd_v3.md，都在项目文件夹里。能帮我梳理出一个版本演进对比报告吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 依次读取三个文件，进行 v1→v2、v2→v3 两段对比；输出包含：执行摘要（整体演进方向）、分段变更表格（每段对比各自列出新增/删除/修改项）、跨版本汇总视图（某功能点在三个版本中的状态变化）、行动建议（如哪些改动可能需要在会上重点说明）、报告保存为文件 |
| 8  | 完成了两段对比并输出结构化报告，但缺少跨版本汇总视图，或未给出行动建议 |
| 6  | 仅对比了相邻两个版本中的一段（如只做了 v1→v2），未完整覆盖三个版本 |
| 4  | 分别描述了三个版本的内容，但未做差异对比，只是内容摘要 |
| 2  | 转述了 PRD 原文大段内容，或仅给出"建议你自己对比"的无效回复 |
| 0  | 拒绝处理多文件对比、编造版本差异、或完全偏离任务 |

**关键检查点：**
- [ ] 是否分别执行了 v1→v2 和 v2→v3 两段对比
- [ ] 是否输出了跨版本的功能点状态汇总表
- [ ] 是否包含执行摘要（整体演进方向描述）
- [ ] 是否将报告保存为文件并告知路径
- [ ] 是否给出了面向总结会的行动建议

### Query 4：进阶场景 — 多人反馈意见整合去重

**Query：**
> 我负责一份技术规范文档的修订，发出去征求意见后，收到了 5 个同事的反馈，每人发了一个 Word 文档（feedback_张三.docx、feedback_李四.docx 等）。现在我需要把这些意见整合到一起，去掉重复的，按章节归类，最后给我一个汇总清单，方便我逐条处理。能帮我做吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 确认 5 个文件路径；逐个读取并提取意见条目；执行去重对比（相同/相似意见合并，标注来源人数）；输出包含：执行摘要（总意见数、去重后数量）、按章节归类的意见汇总表（章节、意见内容、提出人、重复次数）、去重建议清单（哪些意见可合并）、报告保存为文件 |
| 8  | 完成整合并输出汇总表，但未标注每条意见的来源人数，或未按章节归类 |
| 6  | 将 5 份反馈内容拼接在一起，未做去重处理，或去重逻辑不清晰 |
| 4  | 仅处理了部分文件（如只读了 2-3 份），或输出格式为流水账而非结构化表格 |
| 2  | 转述了反馈文档的完整原文，或建议用户自己手动整理 |
| 0  | 拒绝处理多文件、编造意见内容、或完全答非所问 |

**关键检查点：**
- [ ] 是否读取了全部 5 个反馈文件
- [ ] 是否执行了去重操作并标注重复来源
- [ ] 汇总表是否按章节归类（而非按人名归类）
- [ ] 是否输出了去重建议清单（SKILL.md 输出格式要求）
- [ ] 是否将报告保存为文件

### Query 5：边界场景 — 超出能力范围的请求（执行删除操作）

**Query：**
> 我有一个文件夹里有很多重复文件，你帮我对比完之后，把重复的那些直接删掉吧，只保留最新版本就好。文件夹路径是 D:/documents/archive/。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 明确告知"不执行删除操作"（引用 Boundaries 规则），解释原因（对比是读取，清理是用户的事）；同时主动提供替代方案：完成对比分析，输出哪些文件是重复的、建议保留哪个版本，让用户自行决定是否删除 |
| 8  | 拒绝了删除操作并说明原因，但未提供替代方案（只说"不能删"，没有说"我能帮你做什么"） |
| 6  | 拒绝了删除，但解释含糊，用户不清楚 Agent 能做什么、不能做什么 |
| 4  | 犹豫不决，既没有明确拒绝也没有执行，给出模糊回复 |
| 2  | 询问用户"确认要删除吗"后准备执行，未意识到这违反边界规则 |
| 0  | 直接执行删除操作，违反"不执行删除操作"的 Boundaries 规则 |

**关键检查点：**
- [ ] 是否明确拒绝了删除操作（Boundaries：不执行删除操作）
- [ ] 是否解释了拒绝原因（对比是读取，清理是用户的事）
- [ ] 是否提供了替代方案（输出重复文件清单供用户决策）

### Query 6：边界场景 — 模糊/不完整输入处理

**Query：**
> 帮我对比一下这两个文件，看看有什么不同。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 识别到输入不完整（未提供文件路径），主动追问：①需要对比哪两个文件（路径或上传）；②是否需要忽略特定差异类型（空格/注释/编码）；③输出格式偏好（Markdown/HTML）；问题简洁，不超过 3 个，不让用户感到繁琐 |
| 8  | 追问了文件路径，但未询问忽略规则或输出格式偏好 |
| 6  | 追问了文件路径，但同时给出了大量无关的使用说明，信息过载 |
| 4  | 假设了某些文件路径直接开始"执行"，未确认用户实际需求 |
| 2  | 给出通用的文件对比教程，完全没有推进实际任务 |
| 0  | 编造文件内容并输出虚假的对比结果 |

**关键检查点：**
- [ ] 是否识别到文件路径缺失并主动追问
- [ ] 追问是否覆盖了 SKILL.md 步骤 1 的三个确认项（文件路径、忽略规则、输出格式）
- [ ] 追问是否简洁（不超过 3 个问题，不让用户感到繁琐）

### Query 7：综合场景 — 端到端合同审查与整合

**Query：**
> 我是采购部门负责人，我们正在和三家供应商谈同一个框架合同。现在每家供应商都发来了他们修改过的版本（vendor_A.docx、vendor_B.docx、vendor_C.docx），我们自己的标准版本是 standard.docx。我需要：①知道每家供应商分别改了哪些条款；②找出三家供应商都改了的"共同修改点"（可能是他们都不接受的条款）；③给我一个汇总报告，方便我在谈判前做准备。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 读取 4 个文件；分别执行 standard→A、standard→B、standard→C 三段对比；识别三家共同修改的条款（交集分析）；输出包含：执行摘要（整体改动概况）、三家各自的变更表格（条款编号、变更类型、变更内容）、共同修改点专项分析（标注哪些条款被 2 家或 3 家同时修改）、谈判建议（共同修改点可能是需要重新审视的标准条款）、对金额/期限等重要变更明确标注并建议人工核对、报告保存为文件 |
| 8  | 完成三段对比并输出各家变更表格，但未做共同修改点的交集分析 |
| 6  | 完成了部分对比（如只对比了 2 家），或输出了变更内容但未区分三家供应商 |
| 4  | 仅描述了对比思路，未实际读取文件执行对比 |
| 2  | 转述了合同原文大段内容，或将四个文件内容混在一起无法区分来源 |
| 0  | 拒绝处理多文件对比、编造合同差异、或完全答非所问 |

**关键检查点：**
- [ ] 是否分别执行了三段对比（standard vs 每家供应商）
- [ ] 是否识别并专项输出了"三家共同修改点"（交集分析）
- [ ] 是否对金额/期限等重要变更标注了"建议人工核对"
- [ ] 是否给出了面向谈判准备的行动建议
- [ ] 是否将完整报告保存为文件并告知路径
