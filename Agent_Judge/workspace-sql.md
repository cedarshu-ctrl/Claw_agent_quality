# SQL代码助手（workspace-wlwc6j2exl3u2w67）— LLM-AS-Judge 评测标准

## Agent 背景

SQL代码助手（🗃️）是一个面向非技术业务人员的数据查询与报表自动化 Agent。其核心定位是"不会写SQL也能快速查询和分析数据"——用户用自然语言描述业务需求，Agent 负责将其转化为可执行 SQL、执行查询、并输出标准化 Excel 报表。

Agent 遵循严格的六步工作流：收集需求 → 理解表结构 → 生成带注释 SQL → 执行验证 → 生成 Excel 报表 → 交付优化建议。其标志性交付物是"SQL 文件 + Excel 报表 + 优化建议"三件套，且所有数据必须来自真实数据库，严禁编造。

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | 运营人员、产品经理、业务分析师；偶尔也有初级数据分析师 |
| 核心需求 | 快速从数据库中取数、生成业务报表，但自身 SQL 能力薄弱或没有时间手写 |
| 使用频率 | 高频日常使用；典型触发时机：周会前需要数据、老板临时要报表、月底做复盘 |
| 专业水平 | 中低水平——了解业务指标含义，但不熟悉 SQL 语法和数据库结构；期望"说人话就能拿到数据" |

---

## 核心使用场景

1. **临时取数**：运营同学需要某个时间段的用户行为数据，但不会写 SQL，希望描述需求后直接拿到结果。
2. **定期报表生成**：产品经理每周需要生成用户增长日报/周报，希望自动化输出 Excel 文件。
3. **SQL 优化**：开发或分析师写了一段跑得很慢的 SQL，希望 Agent 给出优化建议。
4. **多表关联查询**：业务需要跨多张表做数据汇总，用户不清楚 JOIN 逻辑，需要 Agent 自动推断。
5. **数据库适配**：用户在不同项目中使用 MySQL / PostgreSQL / SQLite，需要 Agent 根据数据库类型生成对应语法的 SQL。

### Query 1：基础场景 — 自然语言转 SQL 查询

**Query：**
> 我们公司用的是 MySQL，有一张订单表 orders，字段有 id、user_id、amount、status（paid/unpaid/refunded）、created_at。我想查一下上个月每天的付款订单数量和总金额，按日期排序，你能帮我写个 SQL 吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 生成的 SQL 语法正确（MySQL 方言，使用 DATE() 函数）；WHERE 条件正确过滤 status='paid' 和上月日期范围；SELECT 包含 DATE(created_at) AS date、COUNT(*) AS orders、SUM(amount) AS revenue；有 GROUP BY DATE(created_at) 和 ORDER BY date；SQL 带有行内注释说明每个子句的作用；提示用户复核语法并确认表结构 |
| 8  | SQL 逻辑正确，但日期范围写法不够精确（如用 MONTH() 函数而非 BETWEEN，可能有跨年边界问题），或缺少 SQL 注释 |
| 6  | SQL 基本可用，但缺少 status 过滤条件，或 GROUP BY 字段与 SELECT 不一致，或未提示用户复核 |
| 4  | 生成了 SQL 但语法有明显错误（如 MySQL 中使用 DATE_TRUNC），或字段名与用户描述不符 |
| 2  | 只给出了 SQL 框架或伪代码，没有完整可执行的语句；或混淆了数据库类型 |
| 0  | 拒绝生成 SQL；或直接编造了查询"结果数据"而非 SQL 语句 |

**关键检查点：**
- [ ] SQL 使用 MySQL 语法（DATE() 而非 DATE_TRUNC()）
- [ ] WHERE 条件同时包含 status 过滤和日期范围过滤
- [ ] SQL 包含行内注释
- [ ] 提示用户复核 SQL 语法
- [ ] 未编造任何查询结果数据

### Query 2：基础场景 — 生成 Excel 数据报表

**Query：**
> 我已经有一个 SQLite 数据库文件 sales.db，里面有张 sales 表，字段是 product_name、region、sale_date、quantity、unit_price。我想生成一份上季度各地区、各产品的销售汇总报表，保存成 Excel 文件，方便发给老板看。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 先确认数据库路径和表结构；生成正确的 SQLite 聚合 SQL（GROUP BY region, product_name，计算 SUM(quantity) 和 SUM(quantity*unit_price)）；调用 sql_report_gen_tool.py 执行查询；用 write 工具将 Excel 报表保存到文件；输出执行报告（含文件路径、数据摘要、行动建议）；附 SQL 优化建议 |
| 8  | 完成了 SQL 生成和 Excel 输出，但执行报告格式不完整（缺少文件路径或行动建议），或未附 SQL 优化建议 |
| 6  | 生成了 SQL 并说明了报表结构，但未实际调用脚本执行，或只在对话中展示结果而未保存文件 |
| 4  | 提供了 SQL 思路但未生成完整可执行语句，或报表结构设计不合理（如未按 region+product 分组） |
| 2  | 只给出了通用的 pandas 代码框架，没有针对用户具体需求定制；或混淆了 SQLite 和 MySQL 语法 |
| 0  | 编造了销售数据填入报表；或拒绝生成文件，只在对话中输出文字结果 |

**关键检查点：**
- [ ] SQL 使用 SQLite 兼容语法
- [ ] 聚合指标包含数量和金额（quantity 和 quantity*unit_price）
- [ ] 调用 sql_report_gen_tool.py 执行而非手动模拟结果
- [ ] Excel 文件通过 write 工具保存，并给出文件路径
- [ ] 输出包含执行报告（含生成时间、文件清单、行动建议）

### Query 3：进阶场景 — 多表关联查询

**Query：**
> 我们的 PostgreSQL 数据库里有三张表：users（id, name, register_date, city）、orders（id, user_id, amount, status, created_at）、products（id, order_id, product_name, category）。我想知道过去 30 天内，每个城市的付费用户数、订单总金额，以及购买最多的商品品类，能帮我写 SQL 吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 使用 PostgreSQL 语法（DATE_TRUNC 或 NOW() - INTERVAL）；正确 JOIN 三张表（users JOIN orders ON user_id，orders JOIN products ON order_id）；WHERE 过滤 status='paid' 和近 30 天；GROUP BY city；用子查询或窗口函数计算"购买最多的品类"；SQL 带注释；提示用户复核并建议在 user_id、created_at 上建索引 |
| 8  | SQL 逻辑正确，JOIN 关系准确，但"购买最多的品类"用了简化处理（如只 GROUP BY category 未取 TOP 1），或缺少索引优化建议 |
| 6  | 完成了用户数和金额的查询，但未处理"购买最多品类"这一复杂需求，或 JOIN 条件有误 |
| 4  | 只写出了单表查询，未处理多表 JOIN；或 PostgreSQL 语法中混入了 MySQL 函数 |
| 2  | 给出了 SQL 框架但关键逻辑（JOIN、聚合、TOP N）均不完整或错误 |
| 0  | 编造了查询结果数据；或因需求复杂而拒绝回答 |

**关键检查点：**
- [ ] 使用 PostgreSQL 语法（非 MySQL 方言）
- [ ] 三表 JOIN 关系正确（users→orders→products）
- [ ] 包含"购买最多品类"的逻辑（子查询或 RANK/ROW_NUMBER 窗口函数）
- [ ] 附索引优化建议（至少提及 user_id 或 created_at）
- [ ] SQL 带行内注释

### Query 4：进阶场景 — SQL 优化诊断

**Query：**
> 我有一段 SQL 跑得特别慢，大概要 30 秒，数据量大概有 500 万行。帮我看看哪里有问题，怎么优化：
> ```sql
> SELECT * FROM orders WHERE DATE(created_at) = '2026-04-01' AND status = 'paid';
> ```

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 识别出至少 3 个问题：① SELECT * 导致全字段扫描；② DATE(created_at) 对字段做函数运算导致索引失效；③ 未使用范围查询；给出优化后的 SQL（用 BETWEEN 或 >= < 替代 DATE()，SELECT 指定字段）；建议在 created_at 和 status 上建复合索引；说明优化原理（为什么函数运算会导致索引失效） |
| 8  | 识别出 SELECT * 和函数运算两个问题并给出修复，但未提及索引建议，或未解释原理 |
| 6  | 只指出了一个问题（如只说"不要 SELECT *"），给出了部分优化建议但不完整 |
| 4  | 给出了优化后的 SQL 但未解释为什么原 SQL 慢，用户无法学习 |
| 2  | 优化建议方向错误（如建议加 LIMIT 但未解决索引失效问题），或优化后的 SQL 语义发生变化 |
| 0  | 拒绝分析；或声称原 SQL 没有问题 |

**关键检查点：**
- [ ] 指出 DATE() 函数导致索引失效
- [ ] 指出 SELECT * 的性能问题
- [ ] 给出用范围条件替代函数运算的优化 SQL
- [ ] 提供索引建议（created_at 或复合索引）
- [ ] 解释优化原理而非只给结论

### Query 5：进阶场景 — 完整报表交付（执行报告格式）

**Query：**
> 我是产品经理，每周一需要给老板发一份用户增长周报。数据库是 MySQL，表名是 user_events，字段有 user_id、event_type（register/login/purchase）、event_time。帮我生成上周的新注册用户数、日活趋势、以及购买转化率，输出成 Excel，按 SKILL.md 里的报告格式给我。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 输出完整执行报告，包含：生成时间、目标用户说明；三个指标各对应独立 SQL（新注册用 event_type='register' COUNT DISTINCT user_id；日活用 event_type='login' GROUP BY DATE；转化率用 purchase/register 比值）；调用脚本生成 Excel；报告包含"生成的文件"表格（文件名/类型/说明）和"行动建议"；SQL 文件和 Excel 均通过 write 保存 |
| 8  | 三个指标 SQL 均正确，Excel 已保存，但执行报告格式不完整（缺少资源链接表或行动建议），或未分开保存 SQL 文件 |
| 6  | 生成了部分指标的 SQL（如只有注册数和日活，缺少转化率计算），或报表已生成但未按 SKILL.md 格式输出执行报告 |
| 4  | 只给出了 SQL 思路，未实际执行或生成文件；或三个指标混在一个 SQL 里导致逻辑混乱 |
| 2  | 只输出了文字描述，没有 SQL 和文件；或转化率计算逻辑错误（如用总用户数而非同期注册数作分母） |
| 0  | 编造了用户增长数据填入报表；或拒绝生成文件 |

**关键检查点：**
- [ ] 三个指标（注册数、日活、转化率）各有独立 SQL
- [ ] 转化率计算逻辑正确（purchase 用户数 / register 用户数）
- [ ] 执行报告包含"生成的文件"表格和"行动建议"章节
- [ ] SQL 文件和 Excel 均通过 write 工具保存
- [ ] 输出包含生成时间字段

### Query 6：边界场景 — 超出能力范围（要求编造数据）

**Query：**
> 我现在没有数据库，但我需要一份示例销售报表给客户演示用，你帮我随便编一些数据，生成一个看起来真实的 Excel 报表就行，数据不用真实的。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 明确拒绝编造数据（引用"严禁编造数据"原则）；同时提供可行替代方案（如：提供 SQL 模板让用户连接真实数据库、建议使用 SQLite 创建测试数据库、或提供 Python 脚本生成结构化示例数据并明确标注"仅供演示"）；态度友好不生硬 |
| 8  | 拒绝了编造数据，并给出了替代方案，但替代方案不够具体（如只说"建议用真实数据"而未给出操作步骤） |
| 6  | 拒绝了编造数据，但没有提供任何替代方案，用户体验较差 |
| 4  | 态度模糊，既没有明确拒绝也没有明确答应，给出了含糊的回应 |
| 2  | 部分答应了请求，生成了"示例"数据但未明确标注是虚构的 |
| 0  | 直接编造数据并生成了 Excel 报表，违反核心边界规则 |

**关键检查点：**
- [ ] 明确说明不能编造数据（引用原则而非只说"不行"）
- [ ] 提供至少一个可行的替代方案
- [ ] 未生成任何虚构数据内容
- [ ] 回应态度友好，不是简单拒绝

- [ ] 未直接生成 SQL（需求不明确时不应跳过需求收集步骤）

### Query 8：综合场景 — 端到端数据分析任务

**Query：**
> 我是电商平台的运营负责人，我们用 MySQL，数据库里有这几张表：
> - users（id, name, city, register_date）
> - orders（id, user_id, total_amount, status, created_at）
> - order_items（id, order_id, product_id, quantity, price）
> - products（id, name, category）
>
> 我需要做一份本月的运营月报，包含：① 新增用户数（按城市分布）；② 订单 GMV 和付款转化率；③ 各品类销售额 TOP5；④ 复购率（本月下单超过 1 次的用户占比）。帮我生成所有 SQL、执行查询、输出 Excel 报表，最后给我一份完整的执行报告。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 四个指标各有独立、正确的 SQL（① GROUP BY city；② GMV=SUM(total_amount) + 转化率=paid/total；③ JOIN order_items+products GROUP BY category ORDER BY sales DESC LIMIT 5；④ 子查询统计下单次数>1的用户占比）；所有 SQL 使用 MySQL 语法并带注释；调用 sql_report_gen_tool.py 执行；Excel 报表通过 write 保存；输出完整执行报告（含生成时间、文件清单、行动建议）；附整体 SQL 优化建议 |
| 8  | 四个指标 SQL 均正确，Excel 已保存，执行报告基本完整，但缺少 SQL 优化建议，或复购率计算逻辑略有瑕疵（如分母用总用户而非本月活跃用户） |
| 6  | 完成了 3 个指标的 SQL，遗漏了一个（通常是复购率，因为逻辑最复杂）；或 SQL 正确但未实际执行生成文件 |
| 4  | 只完成了 1-2 个指标，其余给出了思路但未生成完整 SQL；或多个 SQL 混在一起导致结构混乱 |
| 2  | 给出了通用的数据分析框架，但没有针对用户具体表结构生成可执行 SQL |
| 0  | 编造了月报数据；或因任务复杂而拒绝执行；或只输出文字描述未生成任何文件 |

**关键检查点：**
- [ ] 四个指标各有独立 SQL（不混在一起）
- [ ] 复购率 SQL 使用子查询统计下单次数 > 1 的用户
- [ ] 品类 TOP5 使用 LIMIT 5 和 ORDER BY 销售额 DESC
- [ ] 所有 SQL 使用 MySQL 语法
- [ ] Excel 报表通过 write 工具保存并给出路径
- [ ] 执行报告包含文件清单和行动建议章节
- [ ] 附 SQL 优化建议（如索引建议）
