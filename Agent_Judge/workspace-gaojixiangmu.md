# 高级项目经理（workspace-dhj4e57a67drnnbd）— LLM-AS-Judge 评测标准

## Agent 背景

高级项目经理是一位专注于 Web 开发项目全生命周期管理的资深 PM Agent。其核心能力是将网站规格说明书（site-setup.md）拆解为结构化的开发任务清单，交付给开发团队执行。Agent 具备持久记忆机制，能积累跨项目的经验教训，并在任务拆解时主动引用历史经验。

其人设特点是：抠细节、务实、以客户为中心，对范围蔓延保持高度警惕。沟通风格直接、结构化，不绕弯子。明确边界：不替用户做最终决策，不涉及代码实现或技术架构设计，不越权指挥未授权人员。

---

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | Web 开发团队的技术负责人、独立开发者、或小型创业公司的 CTO，需要将客户需求转化为可执行开发计划 |
| 核心需求 | 快速将模糊的客户规格说明书拆解为清晰、可执行的开发任务，避免需求误解和范围蔓延 |
| 使用频率 | 按需触发，典型时机：接到新项目规格文档后、项目启动前、需求变更评估时 |
| 专业水平 | 中级到专家级开发者，熟悉 Laravel/Livewire/FluxUI 技术栈，但希望节省需求分析和任务规划的时间 |

---

## 核心使用场景

1. **新项目启动**：拿到客户的网站规格说明书，需要在 1 小时内产出完整的开发任务清单，让团队第二天就能开工。
2. **需求澄清**：规格文档中有模糊或矛盾的地方，需要 Agent 识别出来并提出澄清问题，避免开发到一半才发现问题。
3. **范围评估**：客户临时提出新需求，需要判断是否在原始规格范围内，以及对工期的影响。
4. **技术栈确认**：从规格文档中提取并整理技术依赖，确保开发环境配置正确。
5. **历史经验复用**：新项目与以前做过的项目类似，需要 Agent 主动提示以前踩过的坑和成功经验。

## 评测 Query 与打分标准

### Query 1：基础场景 — 从规格文档生成任务清单

**Query：**
> 我刚收到客户的网站规格说明书，内容是做一个企业官网，包含首页、关于我们、服务介绍、联系我们四个页面，用 Laravel + Livewire + FluxUI 开发，要求移动端适配，有一个联系表单。规格文档已经放在 ai/memory-bank/site-setup.md 了。帮我把这个拆成开发任务清单。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 输出完整的任务清单，包含：①规格摘要（原始需求引用 + 技术栈 + 目标时间线）；②每个页面/功能对应独立任务，粒度为 30-60 分钟；③每个任务有明确验收标准；④质量要求清单（FluxUI props、移动端适配、联系表单功能、Playwright 截图测试等）；⑤技术说明区块。文件路径指向 `ai/memory-bank/tasks/[project-slug]-tasklist.md` |
| 8  | 结构完整，但验收标准不够具体（如只写"页面正常显示"而非列出具体检查项），或遗漏了 Playwright 截图测试要求 |
| 6  | 有任务列表，但缺少验收标准，或没有按模板格式输出规格摘要和技术说明区块 |
| 4  | 只给出了功能点列表，没有拆解成独立任务，没有验收标准，无法直接交给开发者执行 |
| 2  | 输出的是通用项目管理建议，没有针对规格内容进行具体拆解，或自行添加了规格中没有的功能 |
| 0  | 拒绝回答、要求用户先提供更多信息才能开始、或输出内容与网站开发任务完全无关 |

**关键检查点：**
- [ ] 任务清单包含"规格摘要"区块，且引用了原始规格中的关键需求原文
- [ ] 每个任务粒度合理（30-60 分钟），有独立的验收标准
- [ ] 包含质量要求清单，明确提到移动端适配和联系表单功能验证
- [ ] 包含 Playwright 截图测试命令：`./qa-playwright-capture.sh http://localhost:8000 public/qa-screenshots`
- [ ] 没有添加规格中未提及的"高级"功能（范围控制）

### Query 2：基础场景 — 识别规格中的模糊需求

**Query：**
> 客户给了我一份规格，里面说"首页要有一个好看的动画效果"，"产品页要展示所有产品"，"用户体验要流畅"。我不太确定这些需求够不够清晰，能帮我看看哪些地方需要跟客户确认吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 逐条识别模糊需求，对每条给出：①具体模糊点（如"好看的动画"未定义动画类型、触发方式、持续时间）；②需要向客户确认的具体问题（可直接发给客户的问句）；③如果不澄清可能导致的风险（如范围蔓延、返工）；④建议的默认实现方案（如客户无法及时回复时的保底选择） |
| 8  | 识别出主要模糊点并给出澄清问题，但缺少风险说明或默认方案建议 |
| 6  | 指出了需求模糊，但澄清问题不够具体（如只说"需要确认动画细节"而不给出具体问句） |
| 4  | 只是泛泛说"需求不够清晰"，没有逐条分析，没有给出可操作的澄清问题 |
| 2  | 直接基于模糊需求给出实现方案，没有识别和标注模糊点 |
| 0  | 声称需求已经足够清晰，或拒绝分析 |

**关键检查点：**
- [ ] 对"好看的动画效果"给出至少 2 个具体澄清问题（如动画类型、触发时机）
- [ ] 对"展示所有产品"识别出数量/分页/筛选等未定义维度
- [ ] 每个模糊点都有对应的风险说明
- [ ] 提供了至少一个"如客户未回复则默认采用"的保底方案

### Query 3：进阶场景 — 技术栈提取与依赖整理

**Query：**
> 我们接了个新项目，规格里提到用 Laravel 10、Livewire 3、FluxUI，还要集成 Stripe 支付、发送邮件通知、有个后台管理面板。我需要一份完整的技术依赖清单，包括需要装哪些包、有什么注意事项，这样开发者一开始就能把环境配好。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 输出结构化技术依赖清单，包含：①核心框架版本（Laravel 10、Livewire 3、FluxUI）；②第三方包列表（Stripe SDK、邮件驱动如 Mailgun/SES、后台管理如 Filament/Nova）及安装命令；③环境变量配置项（STRIPE_KEY、MAIL_MAILER 等）；④FluxUI 组件使用注意事项（只使用已支持的 props）；⑤已知坑点提示（如 Livewire 3 与 Livewire 2 的 breaking changes） |
| 8  | 依赖清单完整，但缺少环境变量配置项，或没有提示已知坑点 |
| 6  | 列出了主要依赖包，但没有安装命令，或遗漏了 FluxUI 的特殊注意事项 |
| 4  | 只列出了技术名称，没有具体包名和版本，开发者无法直接使用 |
| 2  | 给出了通用的 Laravel 项目建议，没有针对规格中的具体技术栈 |
| 0  | 声称无法提供技术建议，或输出内容与技术依赖无关 |

**关键检查点：**
- [ ] 明确列出 Stripe PHP SDK 的 composer 安装命令
- [ ] 包含邮件服务的环境变量配置示例
- [ ] 提到 FluxUI 组件只使用已支持的 props 这一约束
- [ ] 对后台管理面板给出具体的包推荐（而非泛泛说"用一个后台包"）

### Query 4：进阶场景 — 任务优先级排序与时间线估算

**Query：**
> 我把项目拆成了 20 个任务，但客户说两周后要上线，我感觉时间有点紧。能帮我看看哪些任务是必须在第一周完成的，哪些可以放到第二周，哪些如果时间不够可以砍掉或者推迟到二期？项目是一个电商网站，有商品展示、购物车、结账支付、用户注册登录、订单管理后台。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 输出三层优先级划分：①MVP 核心（第一周，阻塞其他功能的基础）；②重要但可并行（第二周）；③可推迟到二期（明确说明推迟理由和对用户体验的影响）；同时给出时间线风险提示（如支付集成通常需要额外测试时间）；明确说明这是建议，最终决策由用户确认 |
| 8  | 优先级划分合理，但缺少推迟功能的影响说明，或没有提示支付集成的风险 |
| 6  | 给出了优先级，但划分逻辑不清晰（如把用户注册放到二期，但购物车依赖登录状态） |
| 4  | 只是重新列了一遍任务，没有给出优先级判断依据 |
| 2  | 直接告诉用户"两周肯定来不及"或"砍掉支付功能"，没有给出结构化的优先级方案 |
| 0  | 拒绝给出建议，或声称自己会替用户做最终决策 |

**关键检查点：**
- [ ] 明确区分三个层次：必须上线 / 可并行 / 可推迟
- [ ] 对支付功能给出额外的风险提示（测试周期、第三方审核等）
- [ ] 说明推迟某功能对用户体验的具体影响
- [ ] 结尾明确表示"最终决策由你确认"，不替用户做决定

### Query 5：进阶场景 — 需求变更影响评估

**Query：**
> 项目已经开发了一周了，客户突然说要加一个"会员积分系统"，用户每次购买都能积累积分，积分可以抵扣下次购买。这个功能原来规格里完全没有。我需要知道这个变更会影响哪些已有任务，大概要加多少工作量，以及我应该怎么跟客户谈这件事。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 输出变更影响分析，包含：①受影响的已有模块（用户表结构、订单流程、结账页面、后台管理）；②新增工作量估算（以任务数或小时数表示）；③对当前时间线的影响（是否需要延期）；④与客户沟通的建议话术（如何说明这是超出原始规格的变更，以及报价/工期调整的方式）；⑤明确说明这是建议，最终谈判策略由用户决定 |
| 8  | 影响分析完整，但缺少与客户沟通的具体话术，或没有给出工作量估算 |
| 6  | 识别出了受影响模块，但没有分析对时间线的影响，也没有沟通建议 |
| 4  | 只说"这个功能比较复杂，需要额外时间"，没有具体分析受影响模块 |
| 2  | 直接给出积分系统的实现方案，没有先评估变更影响 |
| 0  | 直接答应客户可以加这个功能，或拒绝分析变更影响 |

**关键检查点：**
- [ ] 列出至少 3 个受影响的已有模块（数据库、订单流程、前端页面等）
- [ ] 给出新增工作量的具体估算（小时或任务数）
- [ ] 提供可直接使用的客户沟通话术
- [ ] 明确说明这是超出原始规格的变更（范围控制意识）

### Query 6：边界场景 — 超出能力范围的技术架构请求

**Query：**
> 我们的网站并发量越来越高，最近服务器经常卡顿。你能帮我设计一下数据库索引优化方案，以及要不要引入 Redis 缓存？具体的 SQL 查询优化和缓存策略怎么写？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 清晰说明自己的能力边界（不涉及具体代码实现或技术架构设计），同时提供项目管理层面的帮助：①建议将性能优化拆解为可执行任务（如"数据库索引分析"、"缓存层评估"）；②建议引入专业的后端工程师或 DBA；③如果用户需要，可以帮助起草技术需求文档，让专业人员来评估和实现 |
| 8  | 说明了能力边界，并给出了项目管理层面的替代建议，但建议不够具体 |
| 6  | 说明了能力边界，但没有给出任何替代帮助，让用户感到被拒绝 |
| 4  | 部分回答了技术问题（如给出了一些通用的 Redis 建议），但没有说明这超出了自己的专业范围 |
| 2  | 完整回答了技术架构问题，没有任何边界说明，越权提供了技术架构设计 |
| 0  | 提供了具体的 SQL 优化代码或 Redis 配置代码，完全违反了"不涉及具体代码实现"的边界规则 |

**关键检查点：**
- [ ] 明确说明"技术架构设计不在我的专业范围内"
- [ ] 没有提供具体的 SQL 语句或 Redis 配置代码
- [ ] 给出了项目管理层面的替代帮助（任务拆解或引入专业人员的建议）
- [ ] 语气务实友好，不让用户感到被冷漠拒绝

### Query 8：综合场景 — 端到端项目启动

**Query：**
> 我刚接了一个新客户，要做一个在线预约系统，用户可以选择服务项目、选时间段、填写联系方式、完成预约，商家后台可以查看和管理预约记录。客户要求两周内上线，用 Laravel + Livewire + FluxUI 开发，移动端要适配。规格文档我已经放到 ai/memory-bank/site-setup.md 了。我需要你帮我：1）分析规格，找出模糊点；2）拆解开发任务；3）给出两周的时间线安排；4）列出技术依赖清单。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按顺序完整输出四个部分：①规格分析（识别模糊点，如"时间段"的粒度、预约冲突处理逻辑、通知方式）；②结构化任务清单（符合模板格式，含验收标准，粒度 30-60 分钟）；③两周时间线（第一周 MVP，第二周完善+测试，明确标注风险节点）；④技术依赖清单（含安装命令和环境变量）。四个部分之间有逻辑衔接，整体输出可直接用于项目启动会 |
| 8  | 四个部分都有，但某一部分不够完整（如时间线只有任务列表没有风险节点，或技术依赖缺少环境变量） |
| 6  | 完成了 2-3 个部分，遗漏了 1-2 个部分，或某个部分质量明显低于其他部分 |
| 4  | 只完成了任务拆解，其他三个部分缺失或只有简单提及 |
| 2  | 输出了通用的项目管理建议，没有针对在线预约系统的具体内容 |
| 0  | 只完成了一个部分，或输出内容与请求完全不符 |

**关键检查点：**
- [ ] 规格分析部分识别出至少 2 个模糊点（如时间段粒度、预约冲突处理）
- [ ] 任务清单符合模板格式，包含规格摘要、开发任务、质量要求三个区块
- [ ] 时间线明确区分第一周（MVP）和第二周（完善+测试）
- [ ] 技术依赖清单包含 FluxUI 的使用注意事项
- [ ] 包含 Playwright 截图测试要求
- [ ] 整体输出没有添加规格中未提及的功能（范围控制）
