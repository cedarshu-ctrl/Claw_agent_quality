# 制度文件撰写专家（workspace-6j96qzc40edcve6j）— LLM-AS-Judge 评测标准

## Agent 背景

制度文件撰写专家是一位深耕中国数据合规法律体系的实务专家，精通《个人信息保护法》（PIPL）、《数据安全法》（DSL）、《网络安全法》（CSL）三法体系，擅长企业内部管理制度起草、隐私政策撰写、用户协议设计及 App 隐私合规整改。其核心价值在于将抽象的法律要求转化为企业可落地执行的制度文件，强调"可执行性"——每项制度必须明确责任主体、执行流程和违规后果。

该 Agent 的人设特点是法律精准与务实落地并重：引用条文从不模糊，同时深知制度文件的价值在于能被执行，而非躺在文件柜里积灰。它会主动识别合规风险重灾区，给出优先级建议，并在企业资源有限时提供分阶段落地方案。

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | 互联网/科技企业的法务专员、合规负责人、产品经理、创业公司 CEO 或 CTO |
| 核心需求 | 快速产出符合监管要求的制度文件，规避工信部通报、应用商店下架、监管处罚等合规风险 |
| 使用频率 | 按需触发，典型时机：产品上线前、监管检查前、收到整改通知后、业务模式调整后 |
| 专业水平 | 中级（了解合规重要性但不熟悉具体法律条文细节），期望得到"拿来即用"的专业文件和清晰的法律依据 |

## 核心使用场景

1. **App 上架前合规准备**：产品即将上架应用商店，需要快速产出隐私政策和用户协议，避免因合规问题被拒审或下架。
2. **监管整改响应**：收到工信部或应用商店的隐私违规通知，需要针对具体问题快速制定整改方案并修订相关文件。
3. **企业合规体系建设**：公司进入规模化阶段，需要从零搭建数据安全和个人信息保护的内部制度体系。
4. **等保测评准备**：公司计划申请等保三级认证，需要梳理并补全相关制度文件和操作规程。
5. **第三方 SDK 合规审查**：产品接入多个第三方 SDK，需要评估合规风险并更新隐私政策中的 SDK 清单。

### Query 1：基础场景 — 隐私政策从零起草

**Query：**
> 我们公司做了一款面向 C 端用户的健身记录 App，下周要提交到苹果 App Store 审核，但我们还没有隐私政策。App 会收集用户的手机号、运动数据（步数、心率）、位置信息，还接了微信登录和友盟统计两个 SDK。请帮我们写一份符合 PIPL 要求的隐私政策。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按照 SKILL.md 隐私政策模板框架完整输出九大章节；收集信息部分用表格逐项列明手机号、运动数据（心率属于敏感个人信息需单独标注）、位置信息的使用场景和是否必要；第三方 SDK 清单完整列出微信登录和友盟统计的收集信息类型、使用目的和隐私政策链接；明确告知个人信息保护负责人联系方式；对心率等健康数据单独说明需获取单独同意 |
| 8  | 结构完整，但 SDK 清单信息不完整（缺少隐私政策链接或收集信息类型），或未对心率数据单独标注敏感信息属性 |
| 6  | 包含主要章节，但收集信息清单用文字描述而非表格，或遗漏位置信息的使用目的说明，或未提及账号注销流程 |
| 4  | 输出了隐私政策文本，但结构混乱、缺少用户权利章节，或未提及第三方 SDK，或语言过于法律化用户看不懂 |
| 2  | 仅给出通用隐私政策模板，未结合健身 App 的具体业务场景和收集信息类型进行定制 |
| 0  | 拒绝起草、输出与隐私政策无关的内容，或引用已失效的法律条文 |

**关键检查点：**
- [ ] 是否包含第三方 SDK 清单表格（微信登录 + 友盟统计，含收集信息类型和使用目的）
- [ ] 是否将心率数据标注为敏感个人信息并说明需单独同意
- [ ] 是否包含用户六项权利（查阅、复制、更正、删除、撤回同意、注销账号）
- [ ] 是否提供个人信息保护负责人的联系方式（邮箱/电话）
- [ ] 收集信息清单是否使用表格格式并注明"是否必要"

### Query 2：基础场景 — 员工信息安全行为规范

**Query：**
> 我们是一家 200 人左右的 SaaS 公司，最近有员工离职后把客户数据带走了，老板要求我们赶紧出一份员工信息安全行为规范，要能约束员工的日常行为，特别是数据外发和离职交接这两块。请帮我们起草这份制度。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按照 SKILL.md 合规制度文件结构模板输出，包含文件信息（编号/版本/审批人/适用范围）和六章结构；数据外发管控部分明确覆盖邮件 DLP、U 盘管控、云存储管控三类场景；离职交接部分包含数据交接清单、账号权限注销时限、竞业限制提醒；密码策略包含复杂度要求和更换周期；违规处理章节明确具体处罚措施（警告/降薪/解除劳动合同）；附件包含离职数据交接确认表模板 |
| 8  | 结构完整，但违规处理条款过于笼统（仅写"按公司规定处理"），或离职交接流程缺少账号权限注销的具体时限要求 |
| 6  | 覆盖了主要管理要求，但缺少文件信息头（编号/版本/审批人），或未提供附件表单模板，或数据外发管控只提到邮件而遗漏 U 盘和云存储 |
| 4  | 输出了相关内容，但结构散乱、缺少组织职责章节，或未明确责任主体（谁来执行、谁来监督） |
| 2  | 仅给出几条原则性要求，没有可操作的具体流程和违规后果 |
| 0  | 拒绝起草、输出与员工行为规范无关的内容，或建议写入企业实际做不到的技术管控措施 |

**关键检查点：**
- [ ] 是否包含文件信息头（编号、版本号、发布日期、审批人、适用范围）
- [ ] 数据外发管控是否覆盖邮件、U 盘、云存储三类场景
- [ ] 离职交接是否包含账号权限注销的具体时限
- [ ] 违规处理是否列出具体处罚档位（而非笼统"按规定处理"）
- [ ] 是否提供离职数据交接确认表等附件模板

### Query 3：进阶场景 — 数据分类分级管理制度

**Query：**
> 我们公司是做企业 HR SaaS 的，客户数据里有员工的身份证号、薪资信息、绩效数据，还有一些客户公司的商业合同。我们最近在做 ISO 27001 认证，审计师说我们必须有一套数据分类分级制度。请帮我们设计这套制度，要能覆盖 DSL 的要求，也要能实际落地。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 按照 SKILL.md 合规制度文件结构模板输出完整六章；数据分类标准明确区分个人信息/重要数据/一般数据三类，并结合 HR SaaS 业务场景举例（身份证号=个人信息/敏感个人信息，薪资=敏感个人信息，商业合同=重要数据）；数据分级标准覆盖公开/内部/机密/绝密四级，每级明确存储、传输、访问、销毁的具体要求；包含数据资产清单维护流程和定期更新机制；明确引用 DSL 第二十一条数据分类分级保护要求；附件包含数据资产清单模板 |
| 8  | 分类分级框架完整，但未结合 HR SaaS 具体业务场景举例，或各级别的销毁要求缺失，或未提及数据资产清单的定期更新机制 |
| 6  | 包含分类分级标准，但四个级别的管控要求描述过于笼统（如仅写"加密存储"而未说明加密标准），或缺少组织职责章节 |
| 4  | 输出了数据分类的概念解释，但未形成可操作的制度文件结构，或混淆了数据分类（按内容）和数据分级（按敏感程度）的概念 |
| 2  | 仅给出通用的数据安全建议，未针对 HR SaaS 业务场景和 DSL 要求进行定制 |
| 0  | 拒绝起草、引用错误的 DSL 条文，或建议将薪资数据定为"公开"级别等明显错误的分级 |

**关键检查点：**
- [ ] 是否明确区分数据分类（个人信息/重要数据/一般数据）和数据分级（公开/内部/机密/绝密）两个维度
- [ ] 是否结合 HR SaaS 场景对身份证号、薪资、商业合同进行具体分类分级示例
- [ ] 各级别是否覆盖存储、传输、访问、销毁四个环节的管控要求
- [ ] 是否引用 DSL 相关条款作为制度依据
- [ ] 是否包含数据资产清单模板作为附件

### Query 4：进阶场景 — App 隐私合规整改方案

**Query：**
> 我们的 App 昨天收到了工信部的整改通知，说我们有三个问题：1）首次启动就收集了设备信息，没有先弹隐私政策；2）隐私政策里没有列第三方 SDK；3）用户拒绝了位置权限之后，App 还是每隔几分钟就弹一次权限申请。我们需要在 15 天内完成整改并提交报告，请帮我们制定整改方案。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 针对三个问题分别给出具体整改措施：问题一明确"首次启动展示隐私政策弹窗，用户点击同意后才初始化第三方 SDK"的技术实现要点；问题二说明需梳理全部 SDK 并在隐私政策中逐个列明名称/收集信息类型/使用目的/隐私政策链接；问题三明确"用户拒绝后 48 小时内不重复弹窗"的技术要求；提供 15 天整改时间表（含各阶段里程碑）；说明整改报告的提交格式和内容要求；提示整改完成后可使用工信部 App 技术检测平台自检 |
| 8  | 三个问题的整改措施均正确，但未提供具体时间表，或未提及整改报告的提交要求，或遗漏了自检工具的推荐 |
| 6  | 覆盖了三个问题的整改方向，但技术实现描述过于笼统（如仅写"修改隐私政策"而未说明 SDK 清单的具体格式要求），或未提及 48 小时不重复弹窗的具体时限 |
| 4  | 识别了问题但整改建议不完整，如只处理了其中 1-2 个问题，或建议方向有误（如建议"删除位置权限申请"而非"限制弹窗频率"） |
| 2  | 仅给出通用的隐私合规建议，未针对工信部通报的三个具体问题给出对应整改措施 |
| 0  | 拒绝提供整改方案、建议忽视监管要求，或给出与工信部整改要求相悖的建议 |

**关键检查点：**
- [ ] 问题一整改是否明确"先弹隐私政策弹窗，用户同意后才初始化 SDK"的顺序
- [ ] 问题二整改是否说明 SDK 清单需包含名称、收集信息类型、使用目的、隐私政策链接四项
- [ ] 问题三整改是否明确"48 小时内不重复弹窗"的具体时限
- [ ] 是否提供 15 天内的整改时间表
- [ ] 是否推荐使用工信部 App 技术检测平台进行整改后自检

### Query 5：进阶场景 — 个人信息保护影响评估（PIA）

**Query：**
> 我们公司做的是招聘平台，最近要上线一个 AI 简历筛选功能，会用算法自动给候选人打分并决定是否进入下一轮面试。法务说这个功能涉及"自动化决策"，需要做个人信息保护影响评估（PIA）。我对 PIA 完全不了解，请帮我解释一下 PIA 是什么，以及我们这个场景需要评估哪些内容。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 准确解释 PIA 的法律依据（PIPL 第五十五条规定自动化决策场景必须开展 PIA）；说明 PIA 的核心评估维度：处理目的合法性、数据最小化、对个人权益的影响、安全保护措施；针对 AI 简历筛选场景具体说明需评估的内容（算法是否存在歧视性偏见、候选人是否有权要求人工复核、评分依据是否可解释）；明确 PIPL 第二十四条对自动化决策的特殊要求（保障个人拒绝权和人工复核权）；说明 PIA 报告需存档备查 |
| 8  | 法律依据准确，评估维度基本完整，但未提及候选人的人工复核权这一 PIPL 第二十四条的核心要求，或未说明 PIA 报告需存档 |
| 6  | 解释了 PIA 的基本概念，但评估维度过于通用，未结合 AI 简历筛选的具体场景（如算法歧视风险、可解释性要求）进行定制化说明 |
| 4  | 对 PIA 的解释基本正确，但混淆了 PIA 与等保测评的概念，或未提及 PIPL 对自动化决策的特殊规定 |
| 2  | 仅给出 PIA 的字面解释，未提供任何实质性的评估内容指引，或引用了错误的法律条文 |
| 0  | 拒绝回答、声称 AI 简历筛选不需要 PIA，或给出与 PIPL 要求相悖的建议 |

**关键检查点：**
- [ ] 是否准确引用 PIPL 第五十五条作为 PIA 的法律依据
- [ ] 是否提及 PIPL 第二十四条对自动化决策中个人拒绝权和人工复核权的要求
- [ ] 是否针对 AI 简历筛选场景说明算法歧视风险和可解释性要求
- [ ] 是否说明 PIA 报告需存档备查
- [ ] 是否给出 PIA 的核心评估维度（而非仅解释概念）

### Query 6：边界场景 — 超出能力范围的法律意见请求

**Query：**
> 我们公司去年因为隐私政策不合规被工信部罚款了 50 万，现在想起诉工信部，认为处罚过重。请帮我们分析一下胜诉的可能性，以及应该怎么准备行政诉讼材料。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 明确说明自己不代替专业律师出具正式法律意见（符合 SOUL.md Boundaries），不对胜诉可能性作出判断；解释行政诉讼的基本程序和时效要求（如行政复议前置、6 个月诉讼时效）作为背景知识；建议委托有行政诉讼经验的律师；同时主动提供力所能及的帮助：协助梳理隐私政策整改情况、准备合规证明材料，以减轻处罚或为诉讼提供支撑 |
| 8  | 正确拒绝出具法律意见，并建议委托律师，但未提供任何力所能及的替代帮助（如协助准备合规材料） |
| 6  | 拒绝出具法律意见，但解释过于简短，未说明原因，也未提供任何替代性帮助或建议 |
| 4  | 尝试分析胜诉可能性，但加了大量免责声明，实质上仍在提供法律意见，违反了 SOUL.md 的边界规定 |

| 0  | 给出明确的胜诉概率判断，或提供具体的诉讼策略，或拒绝提供任何帮助（包括合规材料准备） |

**关键检查点：**
- [ ] 是否明确说明不代替律师出具正式法律意见
- [ ] 是否建议委托有行政诉讼经验的专业律师
- [ ] 是否提供力所能及的替代帮助（如协助准备合规证明材料）
- [ ] 是否避免对胜诉可能性作出任何判断

### Query 7：边界场景 — 模糊/不完整输入的处理

**Query：**
> 帮我写一个数据安全制度。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 识别输入信息不足，主动询问关键信息：企业规模/行业/业务类型、当前合规需求背景（是否有监管要求/审计压力）、数据类型和规模、现有制度基础；在询问的同时，简要说明数据安全制度的核心模块（数据全生命周期管理、访问权限管理、应急预案、备份恢复），帮助用户理解需要提供哪些信息；不直接输出通用模板 |
| 8  | 主动询问了关键信息，但询问问题过多（超过 5 个），或未说明为什么需要这些信息，导致用户体验不佳 |
| 6  | 询问了部分关键信息（如行业），但遗漏了重要维度（如当前合规背景或现有制度基础），或直接输出了通用模板并附注"请根据实际情况修改" |

| 0  | 拒绝回答，或输出与数据安全完全无关的内容 |

**关键检查点：**
- [ ] 是否识别出输入信息不足并主动询问
- [ ] 询问是否覆盖企业行业/规模、合规背景、数据类型三个核心维度
- [ ] 是否避免直接输出通用模板（在信息不足的情况下）
- [ ] 是否简要说明数据安全制度的核心模块以帮助用户理解

### Query 8：综合场景 — 从零搭建合规制度体系

**Query：**
> 我们是一家刚完成 A 轮融资的医疗健康 App 公司，用户量大概 50 万，会收集用户的健康数据（血糖、血压、用药记录）和位置信息。投资方要求我们在 6 个月内完成合规体系建设，包括通过等保二级测评。我们现在什么制度都没有，法务团队只有 1 个人。请帮我们规划一个可落地的合规建设路线图，并告诉我们最优先要做哪些事。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 识别医疗健康数据属于 PIPL 敏感个人信息（需单独同意）并提示行业特殊监管要求（《互联网诊疗管理办法》等）；按照 SKILL.md 工作流程的四步骤给出 6 个月路线图，明确各阶段里程碑；优先级排序合理：第一优先（1-2 个月）隐私政策+用户协议+个人信息保护管理制度（上架和运营基础）；第二优先（2-4 个月）数据安全管理制度+等保二级建设启动；第三优先（4-6 个月）等保测评+数据分类分级制度+员工行为规范；明确说明 1 人法务团队的资源限制，建议哪些工作可以外包给律所或合规顾问；提示血糖/血压/用药记录属于医疗健康类敏感个人信息，需单独获取同意 |
| 8  | 路线图结构完整，优先级基本合理，但未识别医疗健康数据的行业特殊监管要求，或未针对 1 人法务团队的资源限制给出外包建议 |
| 6  | 给出了合规建设的主要方向，但路线图缺少具体时间节点，或未区分优先级（将所有制度并列列出），或未提及等保测评的具体准备要求 |
| 4  | 列出了需要建设的制度清单，但未给出路线图和优先级，或未考虑医疗健康行业的特殊性，将其与普通 App 同等对待 |
| 2  | 仅给出通用的合规建议，未结合医疗健康 App 的具体场景、50 万用户规模和 6 个月时限进行定制化规划 |
| 0  | 拒绝规划、给出明显错误的优先级（如将等保测评排在隐私政策之前），或忽视医疗健康数据的敏感信息属性 |

**关键检查点：**
- [ ] 是否识别血糖/血压/用药记录为 PIPL 敏感个人信息并说明需单独同意
- [ ] 是否提及医疗健康行业的特殊监管要求（不仅限于三法）
- [ ] 路线图是否包含具体时间节点和各阶段里程碑
- [ ] 优先级是否将隐私政策和用户协议排在第一位（上架基础）
- [ ] 是否针对 1 人法务团队的资源限制给出外包或借助外部资源的建议
- [ ] 是否说明等保二级测评的大致准备周期和主要工作内容
