# 代码文学家（workspace-yw3plsutb1jupnif）— LLM-AS-Judge 评测标准

## Agent 背景

**代码文学家** 📖 是一个专注于代码文档自动化生成的 AI Agent，核心定位是解决开发团队在代码注释和 API 文档编写上耗时、更新滞后、格式不统一的痛点。它支持 Python（Google/NumPy/Sphinx 风格 Docstring）、JavaScript/TypeScript（JSDoc）、Go（godoc）四种主流语言，能够读取真实代码文件、解析函数/类/模块结构，并输出带注释的代码文件和标准化 Markdown API 文档。

Agent 的工作原则是"基于真实代码生成，严禁编造"，所有文档产出必须通过 write 工具落地保存，不能仅在对话中输出。交付物包含：注释后的代码文件、Markdown API 文档、覆盖率报告和行动建议，形成完整的文档化闭环。

## 核心使用场景

1. **存量代码补注释**：接手旧项目或历史代码，函数/类几乎没有注释，需要快速补全 Docstring 以便团队理解和维护。
2. **生成 API 文档**：后端接口开发完成后，需要输出标准化 Markdown API 文档供前端或外部调用方参考。
3. **开源项目文档化**：准备开源一个 Python/JS 库，需要为所有公共 API 生成规范注释和 README 级别的文档。
4. **统一团队文档风格**：团队成员注释风格各异（有的用 Google Style，有的随意写），需要统一重新生成。
5. **文档覆盖率检查**：在 CI/CD 流程中或代码审查前，检查当前代码库的文档覆盖率是否达标（≥80%）。

## 评测 Query 与打分标准

### Query 1：基础场景 — Python 函数 Docstring 生成

**Query：**
> 我有一个 Python 文件 `utils.py`，里面有十几个工具函数，基本没有注释，同事看不懂。文件路径是 `/project/src/utils.py`。帮我用 Google Style 给所有函数加上 Docstring，然后保存回原文件。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 先用 read 工具读取 `/project/src/utils.py`；为每个函数生成符合 Google Style 的 Docstring（含 Args、Returns、Raises 三段）；用 write 工具将带注释的代码保存到文件；输出覆盖率报告（注释函数数/总函数数）和行动建议 |
| 8  | 完成读取、注释生成和文件保存，但部分函数的 Docstring 缺少 Raises 段，或覆盖率报告格式不完整 |
| 6  | 生成了 Docstring 但未调用 write 工具保存文件，仅在对话中输出注释代码；或注释风格不符合 Google Style（如缺少 Args/Returns 标签） |
| 4  | 只对部分函数生成了注释，或注释内容过于简单（仅一行描述，无参数说明） |
| 2  | 未读取实际文件，直接编造了函数内容和注释；或输出的是 JSDoc 格式而非 Python Docstring |
| 0  | 拒绝执行、答非所问，或声称无法读取文件但不给出任何替代方案 |

**关键检查点：**
- [ ] 是否调用 read 工具读取了指定代码文件
- [ ] 生成的 Docstring 是否包含 Args、Returns 段（Google Style 必须项）
- [ ] 是否调用 write 工具将结果保存到文件（不能只在对话输出）
- [ ] 是否输出了文档覆盖率统计（注释函数数/总函数数）

### Query 2：基础场景 — JavaScript JSDoc 注释生成

**Query：**
> 我们前端项目有个 `api.js` 文件，里面全是 async 函数调用后端接口，没有任何注释。路径是 `/frontend/src/api.js`。帮我加上 JSDoc 注释，特别是参数类型和返回值类型要写清楚，方便 IDE 自动补全。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 读取 `/frontend/src/api.js`；为每个 async 函数生成标准 JSDoc（含 `@param {type}` 参数说明、`@returns {Promise<type>}` 返回值、`@throws` 异常说明）；write 保存文件；输出覆盖率报告和建议 |
| 8  | 完成读取、JSDoc 生成和保存，但 `@returns` 类型标注不够精确（如只写 `Promise` 而非 `Promise<Object>`），或缺少 `@throws` 说明 |
| 6  | 生成了 JSDoc 但未保存文件；或注释缺少类型标注（只有描述文字，无 `{type}` 格式） |
| 4  | 只给部分函数加了注释，或注释格式不是 JSDoc（如用了 `//` 单行注释代替 `/** */` 块注释） |
| 2  | 未读取文件，编造了函数签名和注释内容 |
| 0  | 拒绝执行，或输出了 Python Docstring 格式而非 JSDoc |

**关键检查点：**
- [ ] 是否使用 `/** */` 块注释格式（JSDoc 标准格式）
- [ ] 是否包含 `@param {type} name` 格式的参数说明
- [ ] async 函数的 `@returns` 是否标注为 `Promise<type>` 形式
- [ ] 是否调用 write 工具保存文件

### Query 3：进阶场景 — 生成完整 Markdown API 文档

**Query：**
> 我们有一个 Python SDK 库，主要文件是 `/sdk/client.py`，里面有一个 `APIClient` 类和十几个方法。我需要生成一份给外部开发者看的 Markdown API 文档，要有参数表格、返回值说明和代码示例，最后保存到 `/docs/api.md`。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 读取 `/sdk/client.py`；生成的 Markdown API 文档包含：文档标题和生成时间、每个方法的描述段落、参数表格（Name/Type/Description 三列）、Returns 说明、代码示例（```python 代码块）；write 保存到 `/docs/api.md`；输出执行报告（含生成文件路径、覆盖率、行动建议） |
| 8  | 文档结构完整，但代码示例缺失或过于简单（只有函数调用，无上下文）；或参数表格缺少 Type 列 |
| 6  | 生成了 API 文档但格式不规范（无参数表格，只有文字描述）；或未保存文件 |
| 4  | 只生成了部分方法的文档，或文档内容与 TOOLS.md 中的 Markdown API Doc Template 格式差距较大 |
| 2  | 编造了类和方法内容，未基于真实代码文件生成 |
| 0  | 拒绝执行，或只输出了代码注释而非独立的 Markdown API 文档 |

**关键检查点：**
- [ ] Markdown 文档是否包含参数表格（Name/Type/Description 三列格式）
- [ ] 每个方法是否有代码示例（```python 代码块）
- [ ] 是否输出了执行报告（含生成文件路径和覆盖率）
- [ ] 是否 write 保存到用户指定的 `/docs/api.md` 路径

### Query 4：进阶场景 — 多文件项目文档化 + 覆盖率报告

**Query：**
> 我们的后端项目在 `/backend/` 目录下，有 `models.py`、`views.py`、`serializers.py` 三个核心文件，现在要做代码审查，需要先把这三个文件的文档覆盖率提升到 80% 以上。帮我分析现状，然后补全缺失的注释，最后给我一份覆盖率报告。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 依次读取三个文件；统计每个文件的函数/类总数和已有注释数，计算当前覆盖率；为未注释的函数/类生成 Docstring；write 保存三个文件；输出覆盖率报告（含每个文件的前后对比：X/Y 函数已注释，覆盖率从 A% 提升到 B%）和行动建议 |

| 2  | 未读取实际文件，编造了代码结构和注释内容 |
| 0  | 拒绝处理多文件任务，或只输出了一个文件的结果 |

**关键检查点：**
- [ ] 是否读取了全部三个指定文件
