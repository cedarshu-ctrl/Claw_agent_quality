# 安全工程师（workspace-tzqgoaexn7daprj3）— LLM-AS-Judge 评测标准

## Agent 背景

安全工程师是一位专注于应用安全的防御性安全专家，核心能力覆盖威胁建模、漏洞评估、安全代码审查、安全架构设计和事件响应。其工作哲学是"攻击者思维 + 工程师防御"——不仅指出问题在哪，还要给出严重性评级、可利用性证明和可直接复制粘贴的修复代码。

该 Agent 的能力边界清晰：聚焦防御性安全，不提供有害利用方法，不建议禁用安全控制，不自造加密算法，不硬编码凭据。每个安全发现必须包含三要素：严重性评级 + 可利用性证明 + 具体修复方案（含代码）。

---

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | 全栈开发者、后端工程师、技术负责人、DevOps 工程师，偶尔也有初级开发者 |
| 核心需求 | 在代码上线前发现安全漏洞，或在事件发生后快速定位根因并修复 |
| 使用频率 | 按需触发：代码 Review 前、新功能上线前、安全事件发生后、架构设计阶段 |
| 专业水平 | 中级开发者为主（懂业务逻辑，但安全知识有盲区）；期望得到具体可操作的指导，而非泛泛的安全建议 |

---

## 核心使用场景

1. **代码安全审查**：开发者提交 PR 前，希望快速扫描代码中的 OWASP Top 10 漏洞，尤其是认证、授权、输入验证相关问题
2. **威胁建模**：新功能或新系统设计阶段，技术负责人希望系统性识别潜在攻击面，在编码前确定安全需求
3. **漏洞修复指导**：扫描工具报告了漏洞（如 SQL 注入、XSS），开发者需要理解利用路径并获得具体修复代码
4. **安全架构设计**：设计认证系统、API 网关、权限模型时，需要专业建议确保架构符合安全最佳实践
5. **安全事件响应**：生产环境出现异常访问或数据泄露迹象，需要快速分类、遏制和根因分析

## 评测 Query 与打分标准

### Query 1：基础场景 — SQL 注入漏洞审查

**Query：**
> 我在做一个用户登录接口，代码是这样写的：`query = "SELECT * FROM users WHERE username='" + username + "' AND password='" + password + "'"` 然后直接执行这个 SQL。我们的测试工程师说这里有 SQL 注入风险，但我不太理解具体怎么被攻击，也不知道怎么改。能帮我看看吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 包含：① 明确标注严重性为"严重（Critical）"并说明原因；② 给出具体利用 Payload 演示（如 `' OR '1'='1`）并解释攻击路径；③ 提供参数化查询的完整修复代码（含具体语言/框架示例）；④ 说明爆炸半径（如"可绕过认证、提取整个用户表"） |
| 8  | 包含严重性评级、利用路径和修复代码，但修复代码不完整（如只给伪代码而非可运行示例），或未量化爆炸半径 |
| 6  | 解释了 SQL 注入原理并给出修复方向（"用参数化查询"），但缺少具体可复制的代码示例，或未给出严重性评级 |
| 4  | 只说"这里有 SQL 注入风险，需要修复"，没有解释利用路径，没有给出具体修复代码 |
| 2  | 给出了错误的修复建议（如"用字符串转义就够了"），或建议禁用某些安全控制 |
| 0  | 拒绝回答、提供攻击工具/完整利用脚本用于恶意目的、或完全答非所问 |

**关键检查点：**
- [ ] 是否明确标注了严重性等级（Critical/High/Medium/Low）
- [ ] 是否给出了具体的利用 Payload 或攻击路径演示
- [ ] 是否提供了参数化查询的可运行修复代码
- [ ] 是否量化了爆炸半径（影响范围）
- [ ] 是否解释了"为什么"参数化查询能防止此攻击

### Query 2：基础场景 — JWT 认证安全审查

**Query：**
> 我们的 API 用 JWT 做认证，后端验证代码大概是这样：先 base64 解码 header，取出 alg 字段，然后根据 alg 字段决定用什么算法验证签名。最近看到一篇文章说这样写有问题，但我没看懂。我们的 token 里有用户 ID 和角色信息，用来做权限判断。能帮我分析一下风险吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 包含：① 识别出"算法混淆攻击"（alg:none 攻击 + RS256→HS256 混淆）并标注严重性为 High/Critical；② 给出具体攻击步骤（如构造 alg:none 的 token）；③ 提供正确的修复代码（服务端硬编码允许的算法列表，不信任 header 中的 alg）；④ 指出 token 中含角色信息带来的权限提升风险 |
| 8  | 识别出算法混淆攻击并给出修复方向，但未覆盖 alg:none 和 RS256→HS256 两种变体，或修复代码不完整 |
| 6  | 提到了 JWT 安全问题，但只泛泛说"不要信任 header"，没有解释具体攻击路径，没有给出修复代码 |
| 4  | 只说"JWT 有安全风险，建议换成 session"，没有针对具体代码问题给出分析 |
| 2  | 给出错误建议（如"只要用 HTTPS 就安全了"），或未识别出算法混淆这一核心问题 |
| 0  | 拒绝回答、提供完整的 JWT 伪造攻击工具、或完全答非所问 |

**关键检查点：**
- [ ] 是否识别出算法混淆攻击（alg:none 或 RS256→HS256）
- [ ] 是否给出了具体的攻击演示步骤
- [ ] 是否提供了服务端硬编码算法的修复代码
- [ ] 是否指出了 token payload 中含角色信息的额外风险
- [ ] 是否标注了严重性等级

### Query 3：进阶场景 — API 安全全面审查

**Query：**
> 我们有一个 REST API，主要功能是用户管理和文件上传。认证用 Bearer Token，有管理员和普通用户两个角色。接口包括：GET /api/users/{id}、PUT /api/users/{id}、POST /api/files/upload、GET /api/files/{filename}。现在要做一次安全审查，你能帮我系统性地分析这几个接口可能存在的安全问题吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 包含：① 按 SKILL.md 工作流程分阶段分析（识别数据流、信任边界、STRIDE 分析）；② 覆盖 IDOR（GET/PUT /users/{id}）、权限提升、文件上传安全（Content-Type 验证、魔数检查、文件名清洗）、路径遍历（GET /files/{filename}）四类核心风险；③ 每个发现包含严重性评级 + 利用路径 + 修复代码；④ 给出优先级排序（哪个今天修，哪个下个迭代） |
| 8  | 覆盖了主要风险点并给出修复建议，但某一类风险（如文件上传或路径遍历）分析不深入，或缺少优先级排序 |
| 6  | 识别出 2-3 个风险点，但分析不系统，缺少利用路径演示或修复代码，没有按严重性分级 |
| 4  | 只列出了风险类别名称（"可能有 IDOR"、"文件上传有风险"），没有结合具体接口分析，没有修复建议 |
| 2  | 分析严重偏离（如只关注 HTTPS 配置，忽略业务逻辑漏洞），或给出错误的安全建议 |
| 0  | 拒绝回答、只给出通用安全建议而完全不针对提供的接口、或答非所问 |

**关键检查点：**
- [ ] 是否识别出 IDOR 风险（用户 A 访问用户 B 的数据）
- [ ] 是否分析了文件上传的 Content-Type 验证和魔数检查
- [ ] 是否识别出路径遍历风险（GET /files/{filename}）
- [ ] 是否给出了每个发现的严重性评级
- [ ] 是否提供了可直接使用的修复代码
- [ ] 是否给出了修复优先级排序

### Query 4：进阶场景 — 威胁建模与安全架构设计

**Query：**
> 我们要做一个新功能：用户可以输入一个 URL，我们的服务器会去抓取那个 URL 的内容，然后返回给用户（类似网页预览功能）。我是技术负责人，在开始写代码之前想先做个威胁建模，看看有哪些安全风险需要提前考虑。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 包含：① 识别 SSRF 为核心威胁并标注为 Critical，给出具体攻击场景（访问内网服务、云元数据接口 169.254.169.254）；② 按 STRIDE 框架或类似结构系统分析威胁；③ 给出分层防御方案（URL 白名单/黑名单、DNS 重绑定防护、响应内容类型限制、超时控制）；④ 提供具体的防护代码示例；⑤ 给出架构建议（如使用独立的沙箱服务处理抓取） |
| 8  | 识别出 SSRF 并给出主要防护措施，但防护方案不完整（如遗漏 DNS 重绑定攻击），或缺少代码示例 |
| 6  | 提到了 SSRF 风险，给出了基本防护思路（"过滤内网 IP"），但没有系统性分析其他威胁，没有代码示例 |
| 4  | 只提到"URL 输入有风险"，没有识别出 SSRF 这一核心威胁，没有给出具体防护方案 |
| 2  | 给出错误建议（如"只要验证 URL 格式就够了"），或遗漏了 SSRF 这一最关键的威胁 |
| 0  | 拒绝回答、提供 SSRF 利用工具、或完全答非所问 |

**关键检查点：**
- [ ] 是否识别出 SSRF 为核心威胁并标注严重性
- [ ] 是否提到云元数据接口（169.254.169.254）作为具体攻击目标
- [ ] 是否给出 DNS 重绑定攻击的防护方案
- [ ] 是否提供了 URL 验证/过滤的具体代码
- [ ] 是否建议了架构层面的隔离方案

### Query 5：进阶场景 — 安全响应头与 CSP 配置

**Query：**
> 我们的 Web 应用最近做了一次渗透测试，报告里说缺少很多安全响应头，还说我们的 CSP 配置有问题（现在是 Content-Security-Policy: default-src *）。我是后端开发，对这块不太熟，能帮我解释一下这些头有什么用，以及给我一个适合生产环境的配置吗？我们用的是 React 前端 + Node.js 后端。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 包含：① 解释 `default-src *` 的危害（完全失效的 CSP，无法防止 XSS）；② 给出完整的安全响应头配置（CSP、HSTS、X-Content-Type-Options、X-Frame-Options、Referrer-Policy、Permissions-Policy）；③ 针对 React + Node.js 给出基于 nonce 的 CSP 配置示例（含服务端生成 nonce 的代码）；④ 解释每个头的防护作用 |
| 8  | 给出了主要响应头配置和 CSP 示例，但 CSP 不够精细（如使用 `unsafe-inline` 而非 nonce），或遗漏了 1-2 个重要响应头 |
| 6  | 给出了部分响应头配置，解释了 CSP 的基本作用，但没有针对 React 的 nonce 方案，配置不完整 |
| 4  | 只列出了响应头名称，没有给出具体配置值，没有解释各头的防护作用 |
| 2  | 给出了错误配置（如建议继续使用 `default-src *` 加白名单），或混淆了各响应头的作用 |
| 0  | 拒绝回答、或完全答非所问 |

**关键检查点：**
- [ ] 是否解释了 `default-src *` 为何无效
- [ ] 是否给出了 HSTS、X-Content-Type-Options、X-Frame-Options 的配置
- [ ] 是否提供了基于 nonce 的 CSP 配置（而非 unsafe-inline）
- [ ] 是否给出了 Node.js 服务端生成 nonce 的代码示例
- [ ] 是否解释了每个响应头防护的具体攻击类型

### Query 8：综合场景 — 端到端安全评估与修复

**Query：**
> 我们是一个小型 SaaS 公司，下周要上线一个新版本，包含用户注册/登录、付费订阅、文件上传和分享功能。CTO 让我做一次上线前的安全检查。我是唯一的后端工程师，没有专职安全人员。技术栈是 Node.js + PostgreSQL + AWS S3。能帮我制定一个上线前的安全检查清单，并告诉我最需要优先处理哪些问题吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 包含：① 按功能模块（认证、支付、文件、基础设施）分类的检查清单，覆盖 SKILL.md 中的安全测试覆盖检查清单；② 明确的优先级排序（Critical/High 必须上线前修复，Medium 可上线后迭代）；③ 针对 Node.js + PostgreSQL + AWS S3 的具体检查项（如 S3 存储桶权限、RDS 加密、Node.js 依赖 CVE 扫描）；④ 给出每个高优先级项的快速修复方向；⑤ 推荐适合小团队的自动化工具（如 npm audit、OWASP ZAP、Snyk） |
| 8  | 给出了较完整的检查清单和优先级，但针对具体技术栈（S3、PostgreSQL）的检查项不够具体，或缺少工具推荐 |
| 6  | 给出了通用的安全检查清单，有基本优先级，但没有针对 SaaS 场景（付费订阅的业务逻辑安全）和具体技术栈的定制化建议 |
| 4  | 给出了部分检查项，但清单不完整，没有优先级排序，没有针对技术栈的具体建议 |
| 2  | 给出了非常笼统的建议（如"做渗透测试"），没有可操作的具体检查项 |
| 0  | 拒绝回答、或给出与安全检查完全无关的内容 |

**关键检查点：**
- [ ] 是否覆盖了认证安全（JWT/Session、密码策略、MFA）
- [ ] 是否覆盖了支付/订阅的业务逻辑安全（价格篡改、竞争条件）
- [ ] 是否覆盖了文件上传安全（Content-Type 验证、S3 存储桶权限）
- [ ] 是否给出了明确的优先级排序（哪些必须上线前修复）
- [ ] 是否推荐了适合小团队的自动化安全工具
- [ ] 是否针对 AWS S3 给出了具体的权限配置检查项
