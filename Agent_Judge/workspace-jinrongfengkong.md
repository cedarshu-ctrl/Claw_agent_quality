# 金融风控分析师（workspace-z1xihms1ilyg6nvx）— LLM-AS-Judge 评测标准

## Agent 背景

「金融风控分析师」是一个深耕交易欺诈检测与金融风险防控的专业 Agent，核心能力覆盖四大领域：交易欺诈检测（实时规则引擎 + 机器学习模型 + 人工审核三层体系）、反洗钱合规（AML/KYC/STR 全流程）、电信诈骗识别与防控、以及风控数据与征信应用。Agent 精通中国主流支付渠道（支付宝、微信支付、银联）的风控机制，熟悉央行、金融监管总局的监管要求，能够输出结构化的风控规则矩阵、反洗钱监测报告、风控大盘看板等专业交付物。

Agent 的人设是冷静理性、高度警觉、具备攻防对抗思维的风控专家，擅长用数据量化风险、用结构化框架呈现分析建议，同时注重业务平衡——在守住安全底线的同时控制误伤率，保障用户体验。明确边界：不提供具体投资建议、不涉及资金操作指令、不泄露风控规则具体阈值给非授权人员。

## 核心使用场景

1. **风控规则设计**：业务上线新功能（如大额转账、跨境支付），需要快速设计配套的风控规则和阈值
2. **欺诈案例分析**：发现异常交易或新型攻击手法，需要拆解攻击链路并制定针对性拦截策略
3. **反洗钱合规**：季度末需要编制 STR 报告、梳理 KYC 流程，或应对监管检查前的合规自查
4. **风控指标诊断**：误伤率突然上升或欺诈损失率异常，需要快速定位原因并给出优化方案
5. **电信诈骗防控**：用户投诉被骗或公安协查，需要识别诈骗特征、建立预警模型、配合资金追踪

## 评测 Query 与打分标准

### Query 1：基础场景 — 新业务风控规则设计

**Query：**
> 我们公司是做企业收款的，最近要上线一个新功能：允许商户通过 API 批量发起付款，单次最多 500 笔，每笔金额不限。产品经理说下周就要上线，但我们风控这边还没有针对这个场景的规则。你能帮我快速设计一套风控规则吗？主要担心的是套现和洗钱风险。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 输出完整的风控规则矩阵，包含：①分层规则设计（实时规则层 + 模型层），②针对批量付款场景的具体规则（单笔限额分用户等级、批次频率限制、收款方黑名单比对、资金流向分析），③每条规则明确触发条件、排除条件和处置动作，④提示灰度测试要求和上线审批流程，⑤点出套现和洗钱的具体识别特征（如资金快进快出、收款方集中度异常） |
| 8  | 规则设计完整，但缺少排除条件（如已知合法批量业务场景的豁免逻辑），或未提及灰度测试要求 |
| 6  | 给出了部分规则，但结构不完整（缺少分层设计或缺少处置动作），或只给出通用规则未针对批量付款场景定制 |
| 4  | 仅给出原则性建议（"要控制频率"、"要做黑名单"），没有具体的规则条件和阈值参考 |
| 2  | 回答偏离风控规则设计，转而讨论技术架构或产品功能，未解决用户的核心需求 |
| 0  | 拒绝回答，或给出完全错误的风控建议（如建议不设限额直接放行） |

**关键检查点：**
- [ ] 是否输出了结构化的规则矩阵（含触发条件、排除条件、处置动作三要素）
- [ ] 是否针对批量付款场景设计了批次级别的风控规则（不只是单笔规则）
- [ ] 是否识别出套现和洗钱的具体特征（如资金快进快出、收款方集中度）
- [ ] 是否提醒了新规则上线前的灰度测试和审批流程要求
- [ ] 是否区分了不同风险等级的处置动作（拦截/延迟/人工审核）

### Query 2：基础场景 — 风控指标异常诊断

**Query：**
> 我们的风控大盘今天早上显示误伤率从昨天的 0.08% 突然跳到了 0.31%，大量正常用户的交易被拦截，客服电话都被打爆了。我们昨晚 23 点上线了一条新规则，是针对凌晨大额交易的。现在老板在催，你帮我分析一下可能是什么原因，怎么快速处理？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 输出包含：①快速止血方案（立即回滚或临时调整阈值的具体操作建议），②原因分析框架（规则阈值设置过严、排除条件缺失、时区/时间判断逻辑问题等常见原因），③数据排查步骤（如何从日志中定位被误伤的交易特征），④后续优化建议（如何补充排除条件、如何设计灰度测试避免再次发生），⑤用数据量化影响（误伤率 0.31% 意味着什么量级的用户受影响） |
| 8  | 给出了止血方案和原因分析，但缺少具体的数据排查步骤，或未提及如何避免再次发生 |
| 6  | 建议回滚规则，但没有分析根本原因，也没有给出优化方向，只是解决了眼前问题 |
| 4  | 只给出通用的"检查规则配置"建议，没有针对凌晨大额交易规则的具体分析 |
| 2  | 建议继续观察等待，或给出与误伤率诊断无关的建议 |
| 0  | 拒绝回答，或建议忽略误伤问题继续运行 |

**关键检查点：**
- [ ] 是否给出了立即可执行的止血方案（回滚或调整阈值）
- [ ] 是否分析了凌晨大额交易规则可能的具体问题（如排除条件缺失、阈值过低）
- [ ] 是否提供了从日志中定位误伤交易的排查思路
- [ ] 是否提到了对被误伤用户的快速解封通道（申诉流程）
- [ ] 是否给出了防止类似问题再次发生的改进建议

### Query 4：进阶场景 — 新型欺诈手法识别与规则设计

**Query：**
> 最近我们风控团队发现了一个可疑的交易模式：有一批账号，注册时间都在 2 周以内，每个账号每天会收到 3-5 笔来自不同账号的小额转账（每笔 99-499 元），然后在 2 小时内把钱全部转出到同一个收款账号。这个收款账号是一家注册在海南的贸易公司。这是什么欺诈手法？应该怎么拦截？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 输出包含：①准确识别欺诈手法（拆分转账洗钱/蚂蚁搬家，结合新账号特征判断为洗钱通道账号），②攻击链路分析（资金来源→中转账号→汇聚账号→最终流向），③针对性拦截规则设计（新账号收款频率限制、资金快进快出检测、关联账号网络分析），④合规处置建议（是否需要提交 STR、是否需要冻结账号、是否需要报案），⑤提示该模式可能涉及电信诈骗资金转移，建议对接公安部涉案账户数据库核查 |
| 8  | 正确识别欺诈手法并给出拦截规则，但合规处置建议不完整（如未提及 STR 上报要求） |
| 6  | 识别出资金拆分特征，但未能准确判断欺诈类型，或拦截规则过于简单（只说"限制新账号"） |
| 4  | 只描述了现象，没有给出具体的欺诈类型判断和拦截方案 |
| 2  | 将该模式误判为正常业务行为，或给出错误的处置建议 |
| 0  | 拒绝分析，或完全答非所问 |

**关键检查点：**
- [ ] 是否准确识别出"拆分转账"/"蚂蚁搬家"洗钱手法
- [ ] 是否分析了完整的资金链路（来源→中转→汇聚）
- [ ] 是否设计了针对"资金快进快出"的具体检测规则（含时间窗口和金额阈值）
- [ ] 是否建议提交可疑交易报告（STR）并说明理由
- [ ] 是否提示了对接公安部涉案账户数据库的必要性

### Query 7：边界场景 — 模糊/不完整输入的处理

| 分数 | 标准描述 |
|------|---------|
| 10 | 识别出信息不足，主动追问关键信息（业务类型、当前欺诈率数值、主要欺诈类型、现有风控体系情况），同时提供一个结构化的诊断框架（从欺诈识别率、误伤率、欺诈损失率等维度入手），让用户明白需要提供哪些信息才能给出有效建议 |

| 6  | 给出了通用的风控优化建议，但没有针对用户具体情况，也没有追问关键信息 |

| 0  | 拒绝回答，或给出与风控完全无关的内容 |

**关键检查点：**
- [ ] 是否识别出信息不足并主动追问

- [ ] 是否避免了在信息不足时给出不负责任的具体建议

## 附录：评测注意事项

1. **数据真实性**：Agent 输出的数据（如阈值、比例）应与 SKILL.md 中的参考值保持一致，或有合理的业务解释
2. **合规敏感性**：涉及监管要求时，Agent 应引用正确的法规名称（《反洗钱法》《个人信息保护法》《数据安全法》等）
3. **边界一致性**：Agent 拒绝超出边界的请求时，应保持专业友好，并尽量提供替代价值
4. **结构化输出**：高分响应应使用表格、分层标题、具体数值，而非纯文字描述
5. **业务平衡意识**：优质响应在强调安全的同时，应主动考虑误伤率和用户体验影响
