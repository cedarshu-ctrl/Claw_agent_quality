# 后端架构师（workspace-3010ll10ltqof1lk）— LLM-AS-Judge 评测标准

## Agent 背景

后端架构师是一位资深服务端系统专家，专精可扩展系统设计、数据库架构、API 开发和云基础设施。其核心使命是帮助团队构建健壮、安全、高性能的后端系统，覆盖从微服务拆分、数据库 Schema 设计、API 规范制定，到缓存策略、认证授权、监控告警的全链路架构决策。

人设风格简洁直接，先结论后论据，善用结构化输出（架构规范文档、SQL DDL、代码示例），像一个靠谱的技术顾问——有专业深度但不端架子，该纠正就纠正，绝不编造事实。

## 目标用户画像

| 维度 | 描述 |
|------|------|
| 职业/身份 | 后端工程师、技术负责人（Tech Lead）、初创公司 CTO、全栈工程师中的后端侧 |
| 核心需求 | 在新项目启动或系统重构时，快速获得可落地的架构方案、数据库设计和 API 规范，避免踩坑 |
| 使用频率 | 按需触发，典型时机：新功能立项、性能瓶颈排查、技术选型决策、代码评审前的架构确认 |
| 专业水平 | 中级到高级开发者，有一定后端基础，但在系统设计和架构决策上需要专家级指引 |

## 核心使用场景

1. **新系统架构设计**：项目立项阶段，需要确定整体架构模式（微服务/单体/Serverless）、服务拆分方案和技术栈选型
2. **数据库 Schema 设计**：设计核心业务表结构，包括字段类型、索引策略、关联关系和软删除方案
3. **API 规范制定**：设计 RESTful 或 GraphQL 接口，包括路由结构、认证方式、错误码规范和版本控制
4. **高并发场景优化**：针对秒杀、热点数据、大流量场景设计缓存策略、限流方案和数据库读写分离
5. **安全与合规加固**：为现有系统补充认证授权、数据加密、速率限制等安全层设计

## 评测 Query 与打分标准

### Query 1：基础场景 — 用户认证系统设计

**Query：**
> 我在做一个 SaaS 产品，用户量预计初期几千人，后期可能到几十万。现在要设计用户注册登录这块，我想用 JWT，但不太确定怎么处理 token 刷新和安全退出的问题。能帮我设计一下这个认证系统吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 给出完整认证方案：①Access Token + Refresh Token 双 token 机制，明确各自有效期；②Refresh Token 的存储方式（数据库或 Redis 黑名单）；③安全退出的实现（token 撤销/黑名单）；④数据库 users 表 DDL（含 password_hash、软删除字段）；⑤关键安全措施（bcrypt 哈希、HTTPS、HttpOnly Cookie 或 Authorization Header 的选择建议） |
| 8  | 覆盖双 token 机制和安全退出，但缺少具体的 DDL 示例，或未说明 Refresh Token 的存储方案 |
| 6  | 给出了 JWT 基本用法，提到了 token 刷新，但未解决安全退出问题，或安全措施不完整（如未提 bcrypt） |
| 4  | 只讲了 JWT 的基本概念，没有针对用户具体问题（刷新+安全退出）给出可落地方案 |
| 2  | 回答偏离主题，如大篇幅介绍 OAuth 第三方登录而非自建认证，或给出的方案存在明显安全漏洞（如明文存密码） |
| 0  | 拒绝回答、答非所问，或给出的方案包含严重安全错误（如建议不加密存储密码） |

**关键检查点：**
- [ ] 是否明确区分 Access Token（短期）和 Refresh Token（长期）的有效期设置
- [ ] 是否给出 Refresh Token 的服务端存储/撤销方案（而非纯无状态）
- [ ] 是否包含 users 表的 password_hash 字段设计，并说明使用 bcrypt 等哈希算法
- [ ] 是否说明安全退出时如何使 token 失效（黑名单或版本号机制）
- [ ] 是否提到速率限制防暴力破解

### Query 2：基础场景 — 数据库索引优化

**Query：**
> 我们的订单查询接口越来越慢，现在有大概 500 万条订单数据。用户最常用的查询是：按用户 ID 查自己的订单列表（带分页），按订单状态筛选，偶尔按时间范围查。现在没有任何索引，能帮我设计一下索引策略吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 给出完整索引方案：①针对三种查询场景各自的索引设计（复合索引优先于单列索引）；②具体的 CREATE INDEX SQL 语句；③解释索引选择性和最左前缀原则；④提示分页优化（游标分页 vs OFFSET 分页的性能差异）；⑤说明索引的写入开销权衡 |
| 8  | 给出了正确的复合索引方案和 SQL，但未提分页优化或写入开销权衡 |
| 6  | 给出了索引建议，但只是单列索引，未考虑复合索引的查询覆盖，或缺少具体 SQL |
| 4  | 只给出了通用的"加索引"建议，没有针对具体查询场景设计，或索引设计有明显错误（如对低选择性的 status 字段单独建索引） |
| 2  | 建议方向错误，如建议全文索引或不相关的优化手段，或给出的 SQL 语法有误 |
| 0  | 拒绝回答或完全答非所问 |

**关键检查点：**
- [ ] 是否给出针对 `(user_id, created_at)` 的复合索引（覆盖用户订单列表+时间排序）
- [ ] 是否给出具体的 `CREATE INDEX` SQL 语句
- [ ] 是否说明 status 字段低选择性的问题，建议与其他字段组合
- [ ] 是否提到大数据量下 OFFSET 分页的性能问题，建议游标分页
- [ ] 是否提到 `EXPLAIN ANALYZE` 验证索引效果

### Query 3：进阶场景 — 微服务拆分方案

**Query：**
> 我们现在是一个电商单体应用，Node.js + PostgreSQL，大概有用户、商品、订单、支付、库存、通知这几个模块。系统开始出现性能瓶颈，团队也扩大到了 4 个小组。老板想拆微服务，但我不确定怎么拆才合理，拆的粒度怎么把握，服务间通信怎么做。能给我一个完整的拆分方案吗？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 给出完整微服务拆分方案：①按领域边界（DDD）的服务划分建议，明确每个服务的职责边界；②服务间通信策略（同步 REST/gRPC vs 异步消息队列的适用场景）；③数据库拆分策略（每服务独立 DB）；④API 网关设计（路由、认证、限流）；⑤迁移路径建议（绞杀者模式，而非大爆炸重写）；⑥指出哪些模块不建议拆分（如支付和订单的强一致性问题） |
| 8  | 覆盖服务划分、通信策略和迁移路径，但未讨论数据库拆分或分布式事务问题 |
| 6  | 给出了服务划分建议，但通信策略过于简单（只说"用 REST"），未提迁移策略或数据一致性挑战 |
| 4  | 只给出了通用的微服务介绍，没有针对电商场景的具体拆分方案，或拆分粒度明显不合理 |
| 2  | 建议直接大爆炸重写，或给出的拆分方案导致严重的分布式事务问题而未提解决方案 |
| 0  | 拒绝回答，或给出的方案完全不可落地 |

**关键检查点：**
- [ ] 是否给出符合 SKILL.md 模板的服务分解结构（服务名、职责、数据库、API、事件）
- [ ] 是否区分同步通信（REST/gRPC）和异步通信（消息队列）的适用场景
- [ ] 是否提到绞杀者模式（Strangler Fig）作为渐进式迁移策略
- [ ] 是否讨论支付/订单间的分布式事务问题（Saga 模式或 2PC）
- [ ] 是否建议 API 网关统一处理认证和限流

### Query 4：进阶场景 — 高并发秒杀系统架构

**Query：**
> 我们要做一个限时秒杀活动，预计同时在线 10 万用户抢 1000 件商品。现在的系统是普通的 CRUD，直接打数据库肯定扛不住。我需要一个完整的秒杀系统架构方案，要保证不超卖，同时系统不能崩。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 给出完整秒杀架构：①流量削峰方案（消息队列异步处理 + 前端限流）；②Redis 原子操作防超卖（DECR 或 Lua 脚本）；③数据库层的乐观锁或悲观锁选择及理由；④库存预热方案（活动开始前加载到 Redis）；⑤降级熔断策略（库存耗尽后快速返回）；⑥关键数据结构设计（Redis key 设计、订单状态机） |
| 8  | 覆盖 Redis 防超卖和消息队列削峰，但缺少降级熔断或库存预热方案 |
| 6  | 提到了 Redis 缓存和消息队列，但方案不完整，如未解决 Redis 和数据库的数据一致性问题，或未说明如何防超卖 |
| 4  | 只给出了"加缓存"的通用建议，没有针对秒杀场景的具体防超卖和削峰方案 |
| 2  | 方案存在明显漏洞，如只用数据库事务防超卖（在 10 万并发下不可行），或未考虑 Redis 单点故障 |
| 0  | 拒绝回答或给出的方案在高并发下必然超卖 |

**关键检查点：**
- [ ] 是否使用 Redis DECR 或 Lua 脚本实现原子扣减防超卖
- [ ] 是否给出消息队列（RabbitMQ/Kafka）异步处理订单的方案
- [ ] 是否说明库存预热（活动前将库存写入 Redis）
- [ ] 是否提到熔断/降级（库存为 0 时直接返回，不打数据库）
- [ ] 是否给出 Redis key 的设计示例（如 `seckill:stock:{activity_id}`）

### Query 5：进阶场景 — 完整 API 规范设计

**Query：**
> 我要设计一套内容管理系统的 RESTful API，包括文章的增删改查、分类管理、标签系统，还有用户权限（管理员可以操作所有文章，普通用户只能操作自己的）。能帮我出一份完整的 API 设计规范吗？包括路由设计、请求响应格式、错误码、权限控制怎么做。

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 给出完整 API 规范：①符合 REST 语义的路由设计（资源嵌套、HTTP 动词使用）；②统一的请求/响应格式（含 data、meta、error 字段）；③完整的错误码体系（HTTP 状态码 + 业务错误码）；④基于 RBAC 的权限控制方案（中间件实现）；⑤分页参数规范（page/pageSize 或游标）；⑥API 版本控制策略（URL 前缀或 Header） |
| 8  | 覆盖路由设计、响应格式和权限控制，但缺少错误码体系或版本控制策略 |
| 6  | 给出了路由列表，但响应格式不统一，或权限控制方案过于简单（只说"加 token 验证"） |
| 4  | 只给出了部分路由，没有完整的规范文档，缺少响应格式、错误码等关键要素 |
| 2  | 路由设计不符合 REST 语义（如用 GET 做删除），或权限控制方案存在安全漏洞 |
| 0  | 拒绝回答或给出的 API 设计存在严重安全问题（如无认证的删除接口） |

**关键检查点：**
- [ ] 是否给出符合 SKILL.md 模板的 API 代码示例（含 authenticate 中间件）
- [ ] 是否设计统一响应格式（`{ data, meta: { timestamp } }` 或类似结构）
- [ ] 是否给出具体的错误码（HTTP 状态码 + 业务 code，如 `USER_NOT_FOUND`）
- [ ] 是否说明管理员和普通用户的权限区分实现方式（RBAC 中间件）
- [ ] 是否包含速率限制（rate limiting）的配置示例

### Query 8：综合场景 — 端到端系统设计

**Query：**
> 我要从零开始搭建一个在线教育平台的后端，核心功能是：用户注册登录、课程浏览和购买、视频播放进度记录、讲师上传课程。预计初期 1 万用户，一年内可能到 50 万。技术栈我倾向于 Node.js，数据库还没定。能给我一个完整的后端架构方案吗？包括架构设计、数据库选型和 Schema、核心 API 设计、以及需要注意的安全和性能问题？

**打分标准（10 分制）：**

| 分数 | 标准描述 |
|------|---------|
| 10 | 给出完整端到端方案，覆盖：①架构模式选择（初期单体，预留微服务拆分路径）及理由；②数据库选型建议（PostgreSQL 主库 + Redis 缓存，说明理由）；③核心表 DDL（users、courses、enrollments、video_progress，含索引）；④核心 API 路由设计（认证、课程、购买、进度）；⑤安全措施（JWT 认证、支付安全、视频防盗链）；⑥性能考虑（视频 CDN、进度记录的写入优化、课程列表缓存） |
